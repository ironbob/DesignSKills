# 多角度分析协议（模块 C）

> Checklist 第 3 步用。把一本「有故事的书」用角度库切开，找出值得讲的人和事。
> 核心纪律：**逐条适配、连回具体内容、覆盖而非堆砌**。

## 一、加载角度库

读 `references/angle-library.md`。10 个视角是**工具，不是清单**——不要 10 个都往一本书上套。

## 二、逐条适配（适用 / 调整 / 不用）

对每个视角，对照**这本书的具体内容**判：

- **适用** → 这本书里确实有这个视角可挖的人和事（记下：哪个角色/哪段情节）。
- **调整** → 视角方向对，但要看的角度要微调（如"利益博弈"在这本里更偏"名誉博弈"）。
- **不用** → 这本书没有这个维度的素材（直接跳过，别硬凑）。

> 反模式：照念视角定义当结论（"本书体现了人物命运"）。必须落到**具体的人和事**。

## 三、连回具体内容（防空讲）

每个候选选题点必须答得出：
- 讲的是**谁**（人物）/ **哪件事**（情节）？
- 书里**哪一段**支撑这个角度？
- 这个角度让人**意外/共鸣**的点在哪？

答不出的，丢弃。这是"选题粒度 ≠ 目录复述"的保证（见 `selectability.md`）。

## 四、覆盖约束

- 最终选题要**分散**在 ≥ **3 个不同视角**上（门 `O-COVERAGE` 硬卡）。
- 全挤在"人物命运"一个维度 = 没用透角度库。
- 优先纳入 `counter-intuitive` / `then-now` / `interest-game`（钩子倾向高/最高，爆款潜力大）。

## 五、用户思路优先

若第 1 步用户给了思路（`user_angle_direction ≠ 无`）：
- 用户指定的方向**优先**，先用对应视角深挖；
- 再用其余视角**补充**用户没想到的角度（这是 skill 的增量价值）；
- 不要无视用户思路另起炉灶。

## 六、产出

候选选题点先以**粗粒度**列出（视角 + 谁的事 + 一句钩子方向），交给第 4 步做可讲性筛选与粒度收敛（3-8 个），再进第 5 步填完整 topic。
