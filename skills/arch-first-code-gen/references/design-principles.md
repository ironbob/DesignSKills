# 设计原则库：架构确认的推理依据

> 配合 `arch-first-code-gen` 的 Checklist 第 2 步（模块 B 前置）。这是**架构确认的推理依据**——确认每个角色/拆分时，要说得出「依据哪条原则」。原则不是 checklist 逐条对照打分，而是**每处拆分决策的 why**。
>
> **关键**：原则是推理工具，不是合规清单。用它解释「为什么这么拆」，而不是「凑齐五条 SOLID」。

---

## 一、为什么先加载原则库

PRD 强制：每个角色标注「**业界做法依据 + 所依据的设计原则**」。如果拆分说不出依据，就是「凭感觉设角色」——这正是本 skill 要取代的反模式。先加载这份库，确认角色时随手引用，让设计可追溯。

---

## 二、SOLID（面向对象设计五原则）

| 原则 | 一句话 | 它逼你做什么拆分决策 |
|---|---|---|
| **SRP**（单一职责） | 一个类/模块只有一个变化的理由 | 一个角色承担多个不相关职责 → 拆。问「这个角色会因为哪几类需求变化？」多于一个就拆 |
| **OCP**（开闭） | 对扩展开放、对修改关闭 | 加新策略要改老代码 → 抽接口/策略，新行为新增实现不改既有 |
| **LSP**（里氏替换） | 子类能无感替换父类 | 继承/实现是否真能替换？不能就别用继承，改组合 |
| **ISP**（接口隔离） | 不被迫依赖用不到的方法 | 胖接口 → 按调用方需要拆小接口，调用方只依赖它用的 |
| **DIP**（依赖倒置） | 依赖抽象，不依赖具体 | 上层（controller/service）依赖下层具体实现 → 反过来都依赖抽象（接口），具体可替换、可测试 |

> **拆分时怎么用**：DIP 决定**依赖方向**（上层不直接 new 下层具体类，靠接口/注入）；SRP 决定**粒度**（一个角色一个职责）；ISP 决定**接口胖瘦**；OCP 决定**扩展点**。

## 三、DDD（领域驱动设计）

### 战略设计（划定边界）

| 概念 | 一句话 | 它逼你做什么拆分决策 |
|---|---|---|
| **限界上下文**（bounded_context） | 一个模型在一个边界内有一致含义 | 跨域复用同名概念（如「订单」在交易域 vs 物流域含义不同）→ 各自上下文，不强行共用模型 |
| **上下文映射**（context_mapping） | 上下文之间如何集成（防腐层/共享内核/客户-供应商…） | 跨上下文调用 → 显式定集成方式，常加防腐层（ACL）隔离 |

### 战术设计（领域角色 — PRD 强制的「领域角色」来源）

| 领域角色 | 一句话 | 它逼你做什么拆分决策 |
|---|---|---|
| **聚合根**（aggregate） | 一致性边界 + 对外唯一入口 | 把强一致的业务规则收进聚合根，外部只持有根的引用；不允许跨聚合直接改内部 |
| **实体**（entity） | 有唯一标识、有生命周期 | 区分「有身份的对象」与「纯值」；行为跟数据放一起，不是贫血对象 |
| **值对象**（value_object） | 无身份、不可变、按值判等 | 库存、金额、地址这类 → 值对象，避免基本类型满地（如用 `Money` 而非 `int`） |
| **领域服务**（domain_service） | 不属于任一实体的领域操作 | 一个操作涉及多个聚合、放哪个实体都不合适 → 领域服务 |
| **领域事件**（domain_event） | 领域中发生的事实（过去时） | 需要解耦/通知/审计的关键业务动作 → 发事件，订阅方各自处理 |

> **分层角色 vs 领域角色**：分层角色（Controller/Service/Repository）是**技术分层**；领域角色（聚合/实体/值对象…）是**业务建模**。两者正交——一个 `OrderAggregate`（领域角色=聚合根）住在 domain 层。PRD 要求**两类都过一遍**。

## 四、通用设计原则

| 原则 | 一句话 | 它逼你做什么拆分决策 |
|---|---|---|
| **高内聚低耦合**（high_cohesion_low_coupling） | 相关的放一起，不相关的别缠 | 内聚低（一个模块里东西不相关）→ 拆；耦合高（改一处牵连多处）→ 引入抽象解耦 |
| **依赖方向**（dependency_direction） | 依赖单向、指向稳定方 | 画出依赖箭头，不许成环、不许下层反向依赖上层（架构门可结构性查） |
| **关注点分离**（separation_of_concerns） | 不同关注点不同模块 | 协议适配/业务编排/持久化/横切（日志/鉴权）分开，不混在一个角色里 |
| **Tell-Don't-Ask**（tell_dont_ask） | 让对象做事，别问一堆再替它做 | 取出一堆字段在外部算 → 改成让对象自己提供行为（行为归对象，减少依恋情节） |

## 五、怎么在角色确认里引用

确认每个角色时，填两栏（契约源 `industry_basis` + `design_principles`）：

1. **业界做法依据**：这个角色来自什么通行做法？（如「DDD 聚合根」「MVC Controller」「Repository 模式（PoEAA）」）——对应 `standard-practices/` 按栈库。
2. **所依据的设计原则**：为什么这么拆，依据哪条原则？（如「SRP：只做 HTTP 编排」「DIP：依赖 OrderService 抽象」）——从本库取。

> **反模式**：写「符合 SOLID」「设计良好」这种空泛话。要具体到「SRP：只负责 X」「DIP：依赖 Y 抽象而非 Z 实现」。
