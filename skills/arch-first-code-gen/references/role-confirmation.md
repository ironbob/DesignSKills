# 角色确认法（模块 B 核心）

> 配合 `arch-first-code-gen` 的 Checklist 第 3 步。这是 skill 的**心脏**：参考按栈标准做法库 + 设计原则，**几轮讨论**确认角色（分层 + 领域）/ 职责 / 依赖，产出「角色职责清单」。**未确认不编码（HARD-GATE）。**
>
> 模块 B **只定角色/职责/依赖**——不画图、不写接口（那些是模块 D 的事）。

## 一、确认什么（PRD §4-B）

每轮讨论收敛到一份「角色职责清单」，每行：

| 角色 | 类型 | 层 | 领域角色 | 职责（单一） | 依赖 | 业界做法依据 | 设计原则 |

- **类型**：分层角色（layer）/ 领域角色（domain）。
- **层**：controller/service/repository/domain/infrastructure/...
- **领域角色**：aggregate/entity/value_object/domain_service/domain_event/null。
- **职责**：动词开头，单一（SRP）。
- **依赖**：依赖哪些角色（依赖方向，DIP）。
- **业界做法依据 + 设计原则**：PRD 强制——每角色说得出「来自什么业界做法」「依据什么原则」。

## 二、确认流程（几轮讨论）

1. **从标准做法库取候选**：按栈加载 `standard-practices/<stack>.md`，把它的角色目录当候选池。
2. **逐条判适用性**：对照具体需求，每个候选角色判「**适用 / 调整 / 不适用（+理由）**」。只采用过判断的，不照搬。
   - 例：需求是纯 CRUD 无复杂领域逻辑 → 判「不用聚合根，只 controller/service/repository，理由：无跨实体一致性约束」。
3. **定职责 + 依赖**：每个采用的角色写一句话职责 + 依赖谁；检查依赖方向单向（架构门可查）、单一职责（SRP）。
4. **分层 + 领域两类都过**：先过分层角色，再过领域角色；任一类不用要说明理由，不静默漏。
5. **标注依据**：每角色填 `industry_basis` + `design_principles`（从 `design-principles.md` 取）。

## 三、收敛与回退（PRD 未决项的 skill 层约定）

- **收敛信号**：角色清单稳定（增删趋于 0）、依赖方向无环、每角色有依据 → 呈现完整清单请用户确认。
- **几轮的边界**：通常 2-4 轮足够；超过仍未收敛，说明需求边界可能不清，提示回去收紧范围（模块 A）。
- **用户否决某角色**：回到第 2 步重判——否决后其职责归谁？依赖怎么调整？更新清单重确认。
- **不画图、不写接口**：若讨论滑向「画个架构图」「定义接口签名」，拉回「模块 B 只定角色/职责/依赖；图与接口在模块 D 统一生成」。

## 四、与业务流程梳理的衔接（P1）

角色确认基本完成后，按 `business-process.md` 梳理**主流程 + 异常分支**（P1），为模块 D 流程图 + 模块 E 覆盖门打底。流程每步要能映射到参与的角色。

## 五、角色职责清单 → 契约源

确认后，清单直接映射到 `design-contract.json` 的 `roles[]`（见 `design-contract-schema.md §三`）：每个角色一个对象，含 id(`ROLE-[LD]<n>`) / name / role_kind / layer / domain_role / responsibility / depends_on / industry_basis / design_principles / code_units。

> `code_units`（代码文件路径）在模块 C 编码后回填；模块 B 阶段可留空或填预期路径，交付前应真实存在，`validate_gate.py` 可辅助发现悬空角色。

## 六、反模式

| 反模式 | 正确做法 |
|---|---|
| 凭「感觉要个 Service」设角色 | 每角色标注业界做法依据 + 设计原则 |
| 只设计分层角色，漏领域角色 | 分层 + 领域两类都过，不用说明理由 |
| 模块 B 就画图/写接口 | 只定角色/职责/依赖；图与接口留模块 D |
| 角色职责写成「管理订单」（含糊） | 动词开头、单一：「接收下单请求、校验入参、编排」 |
| 依赖方向成环 / 下层依赖上层 | 单向、指向稳定方（DIP/dependency_direction） |
| 业界依据写成「符合业界标准」（空泛） | 具体到「MVC Controller（Spring @RestController）」「DDD 聚合根」 |
