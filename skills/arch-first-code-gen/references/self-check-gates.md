# 原则导向自检（模块 E — 三道门）

> 配合 `arch-first-code-gen` 的 Checklist 第 9 步。模块 E 不是机械卡关，而是**原则导向复核**：先判断架构原则与代码设计原则是否被真实遵守，再用架构门 / 日志门 / 覆盖门提供结构性证据。脚本结果是证据，不替代设计判断。
>
> `scripts/validate_gate.py` 可辅助执行：读 `design-contract.json`（契约源）+ `<feature>-arch.md`（渲染）+ 仓库根（文件存在性），输出三门结论 + go/no-go + 问题清单。它只能做结构性近似；语义判断仍要由 Codex 依据设计原则复核，并在缺口里诚实说明。

## 一、三道门（PRD §4-E）

### 架构门（architecture）
- **结构性查（脚本证据）**：
  - 每个角色 `code_units` 的文件**真实存在**（角色↔代码覆盖）。
  - 每个角色 `depends_on` 的 id 都能解析到已定义角色（无悬空依赖）。
- **原则复核（核心）**：职责是否真单一（SRP）、依赖方向是否合理（DIP/dependency_direction）、领域角色是否承载领域规则（DDD/Tell-Don't-Ask）、是否存在跨层或横切逻辑污染。脚本查不到的语义项要明确写入 `gate.notes` /「已知缺口」，**不假装查了**。

### 日志门（logging）
- **结构性查**：按栈日志关键字（JVM `log.`/`logger.`/`LOGGER.`；C++ `spdlog::`；Python `logger.`/`log.`）统计每个代码单元的日志覆盖。
- **判定**：代码单元中**有关键节点却无任何日志关键字**的，按比例告警。这是「日志关键字是否出现在关键位置」的结构性近似，不是绝对质量结论。
- **原则复核（核心）**：入口/出口/异常/外部调用是否围绕业务可观测性打点，ERROR 是否带上下文，日志是否服务排障而非噪音。结构查不全的，LLM 自检 + 缺口。

### 覆盖门（coverage）
- **结构性查（脚本证据）**：
  - 业务流程每步 `code_refs` 文件**存在**。
  - 每步 `roles` 都是已定义 role id。
  - 每步有 `doc_ref`。
  - **契约↔文档对账**：文档角色职责表的角色集合 == 契约 roles 的 name 集合；`roles_count`/`process_steps`/`verdict` 一致。
- **原则复核（核心）**：流程覆盖要证明职责拆分真的落地：每一步由合适角色承担，异常分支没有被塞进错误层，文档解释了为什么这么分。三者（流程↔代码↔文档）对不上通常是真问题；若脚本因路径/示例/近似能力误伤，说明原因并登记缺口。

## 二、判断规则

- `gate.verdict` 应表达**原则复核后的总体判断**，并参考脚本三门输出。
- 脚本发现的真实结构问题（角色无代码、依赖悬空、流程代码落点不存在、文档角色漂移）通常应修复后再交付。
- 脚本 no-go 但判断为近似误伤时，不要机械降阈值；在 `gate.issues[]` / `gate.notes` / 文档「已知缺口」写清：问题是什么、为什么不构成当前设计阻断、后续如何处理。
- `go` 不等于“完美”，只表示本次 feature 的架构原则、代码设计原则和必要结构证据已达到交付要求。

## 三、结构性 vs 语义性（诚实划界）

| 能力 | 谁查 | 强度 |
|---|---|---|
| 角色有对应代码文件 / 依赖可解析 / 流程有代码落点 | 机器（`validate_gate.py`） | 结构性证据 |
| 日志关键字出现在代码单元 | 机器（按栈匹配） | 覆盖率近似 |
| 文档角色 == 契约角色 / 计数一致 | 机器（对账） | 结构性证据 |
| 职责是否真单一 / 依赖是否真合理 / ERROR 是否真带上下文 | LLM 语义自检 | 核心设计判断，必要时登记缺口 |

> **纪律**：机器查不到的，在 `gate.notes` + 文档「已知缺口」诚实标注「辅助校验未覆盖（语义项）」，**不把语义判断伪装成机器已验证**。

## 四、issue 结构（契约源 `gate.issues[]`）

| 字段 | 说明 |
|---|---|
| `gate` | architecture / logging / coverage |
| `severity` | critical / major / minor |
| `role_or_step` | role id 或 `step:<n>` |
| `problem` | 问题描述 |
| `evidence` | 文件:行 / 方法 / 或「辅助校验未覆盖（语义项）」 |

## 五、自检不做什么（诚实）

- ❌ **不做 AST / 静态依赖分析**（那是 `arch-quality-eval` 的能力；生成侧自检做不到，PRD §6）。
- ❌ **不逐角色硬卡编码过程**（设计契约是软引导；原则复核在阶段末完成）。
- ❌ **不假装语义项被机器验证**（职责/合理性结构查不到，登记缺口）。
- ❌ **不做 lint / 代码风格**（与评估侧同口径，PRD §5）。
- ❌ **不做 CI 卡关**（开发时质量保障，PRD §5）。

## 六、发现问题怎么办

no-go 或原则复核未通过不是失败信号，是「还没达标」的反馈：

- 架构门 no-go（角色无代码 / 依赖悬空）→ 回模块 C 补代码 / 修契约依赖。
- 日志门 no-go（关键节点无日志）→ 回模块 C 补日志。
- 覆盖门 no-go（流程/角色对不上）→ 回模块 B/C/D 对齐流程↔代码↔文档。
- 原则复核发现职责混杂 / 依赖反向 / 领域贫血 → 回模块 B 重新确认角色，或回模块 C 调整代码。

修完按需重跑三道门；交付前要能说清「哪些原则被遵守、哪些结构证据支持、哪些缺口仍存在且不阻断本次交付」。

## 七、反模式

| 反模式 | 正确做法 |
|---|---|
| 把生成侧辅助校验当 AST 强度 | 结构性机器查 + 语义 LLM 自检，诚实登记缺口 |
| 把语义项伪装成机器已验证 | 语义项标「辅助校验未覆盖」登记缺口 |
| 脚本 no-go 就降阈值蒙混过去 | 判断是否是真设计问题；真问题修，近似误伤登记缺口 |
| 校验只查「能跑」 | 三道门提供架构覆盖/日志覆盖/流程覆盖证据，不止能跑 |
| 覆盖门只查流程不查文档 | 流程↔代码↔文档三者对得上 + 契约↔文档角色对账 |
