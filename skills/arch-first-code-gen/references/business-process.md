# 业务流程梳理（模块 B P1 / 模块 D / 模块 E）

> 配合 `arch-first-code-gen` 的 Checklist 第 3 步(P1)。角色确认后梳理**业务流程主线 + 异常分支**，为：① 模块 D 流程图、② 模块 E 覆盖门（流程↔代码↔文档三者对得上）打底。覆盖门是用户重视的硬关，流程梳理质量直接决定它能不能过。

## 一、梳理什么（PRD §4-B P1）

- **主流程**：从触发到达成的核心步骤序列（如「下单」：接收请求 → 校验库存 → 创建订单聚合 → 持久化 → 发事件 → 返回）。
- **异常/分支**：每步的关键异常/分支**至少标注**（如「库存不足 → 抛异常」「支付失败 → 回滚」）。
- **角色映射**：每步涉及哪些角色（role id），让流程与角色清单对得上。

## 二、流程步骤结构（契约源 `business_process[]`）

每步：

| 字段 | 说明 |
|---|---|
| `step` | 序号（1 起，递增） |
| `name` | 步骤名（动词短语，如「校验库存并创建订单聚合」） |
| `roles` | 参与本步的 role id（须有效，如 `["ROLE-L02","ROLE-D01"]`） |
| `code_refs` | 本步代码落点（仓库根相对路径，可带 `:方法`）；覆盖门校验**文件存在** |
| `doc_ref` | 本步在架构文档的体现（如「业务流程图-步骤2」）；覆盖门交叉对账 |
| `exception` | 本步异常/分支（至少标注；无则 `null`） |

## 三、梳理纪律

1. **主流程优先**：先理清主路径，再补异常分支；别一上来铺所有边界条件。
2. **步骤可映射到代码**：每步都要有明确代码落点（code_refs）。若某步找不到代码 → 要么漏了角色/代码，要么这步不该在流程里。
3. **异常至少标注**：关键异常路径标出（不要求穷举所有边界），覆盖门 + 模块 D 流程图要覆盖「主流程 + 关键异常分支」。
4. **角色有效**：每步 `roles` 必须是已确认的 role id（覆盖门可辅助发现），不能引用未定义角色。

## 四、与覆盖门的关系（模块 E）

覆盖门要求：**流程每步 ↔ 代码(code_refs 文件存在) ↔ 文档(doc_ref) 三者对得上**。所以梳理时就要保证：

- 每步有 `code_refs`（模块 C 编码后回填真实路径）。
- 每步有 `doc_ref`（模块 D 生成流程图时对齐）。
- 每步 `roles` 有效。

> 梳理潦草（步骤无代码落点 / 角色未定义）→ 覆盖门直接 no-go。这是把「流程想清楚」前置到模块 B 的价值。

## 五、反模式

| 反模式 | 正确做法 |
|---|---|
| 只写主流程，无视异常 | 关键异常分支至少标注 |
| 步骤无代码落点 | 每步要有明确 code_refs |
| 步骤引用未定义角色 | roles 必须是已确认 role id |
| 流程步骤与文档对不上 | doc_ref 在模块 D 流程图里对齐 |
| 穷举所有边界条件淹没主流程 | 主流程优先，关键异常标注即可 |
