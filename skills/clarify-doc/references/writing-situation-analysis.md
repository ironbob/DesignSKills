# 沟通场景分析：用专业文体与业界做法审视写作意图

> 配合 clarify-doc skill 的 Checklist 第 2 步。
> 目的：在澄清写作意图**之前**，先用专业文档类型的通行做法做一遍基本分析，识别用户原始表述里的**盲区**（漏掉的关键信息点）和**风险**（选错文体/对象/动作），让后续澄清建立在专业判断之上，而非盲目接受用户的初始表述。
> 与 `doc-blueprint` 的 `doc-type-library/` 是互补关系：本文件管"该选哪类 + 专业版必备什么"（选型与盲区诊断），那里管"这类文档怎么搭结构"（落地模板）。两者共用同一套文档类型本体。

---

## 一、为什么先做场景分析

用户的一句模糊题目往往只包含"他想到的"，而漏掉了这类文档**几乎都要传达的关键信息**，也可能**选错了文体**（把事故复盘写成项目宣传稿、把决策请示写成技术说明书）。若直接问"你还想写什么"，用户多半答不上来——他不知道自己漏了什么。

场景分析的价值：用专业视角**主动指出**这类文档的必备信息点与常见陷阱，把它们变成可讨论的具体选项，再让用户取舍。这才是"澄清"而非"记录"。

---

## 二、分析维度

对该写作题目所属的**沟通场景**，逐项过一遍。每项都要有结论（"有 / 无 / 不适用"）。

### 1. 沟通本质
这篇文档**本质**在做什么？是【说服决策】【同步进展】【总结复盘】【指导操作】【记录约定】【传递知识】【预警风险】中的哪一种？不同本质对应不同文体。
> 例：同样是"写个东西讲这次事故"——对内复盘要根因+改进；对外公告要影响+补偿；两者文体完全不同。

### 2. 读者与期望动作
谁读？读完要做什么（决策 / 执行 / 知晓 / 说服）？**动作决定一切**：要人拍板 → 给结论+选项+建议；要人执行 → 给步骤+前置+回滚；只要人知晓 → 给要点+影响。
> 例：写给领导审批预算 → 动作是"批准"；那文档必须让他在 3 分钟内看清"要多少钱、回报是什么、不批的代价"。

### 3. 文体选择（最关键）
这种沟通场景，业界通行用哪类文档？用户原始表述是否选对？
> 见第四节决策树。

### 4. 业界标杆做法
该文体公认做得好的组织/模板，在关键环节怎么处理？有什么被验证过的结构？
> 例：Google design doc、Amazon 6-pager（叙事备忘录，禁 PPT）、阿里/字节的复盘模板（时间线+根因+Action+Owner+DDL）、AWS Runbook（触发条件+步骤+可复制命令+回滚+升级）。

### 5. 常见陷阱与反模式
这类文档**常踩的坑**：选错文体、写给错对象、漏掉关键信息点、堆砌而无关点、只说"是什么"不说"所以呢"。
> 见第五节每类文体的"常见缺陷"。

## 三、如何呈现给用户

场景分析必须把发现**结构化地呈现给用户**，作为澄清的起点：

```
基于这种场景（<沟通本质>）的专业与业界做法，我先梳理几点：

【沟通本质】这篇本质是 <说服决策/同步/复盘/…>，关键动作是 <读者读完要做什么>。
【专业文体】这种情况业界通行写 <文档类型>，专业版必须传达 <A / B / C>。
【盲区】你的表述里暂未提到 <D / E>——这些通常是这类文档的关键，需要吗？
【风险】这类文档常踩的坑：<陷阱>，要注意避免。

请看这些是否符合你的预期，哪些要写、哪些先不写？
```

呈现后，把用户的回应作为 Checklist 第 3-5 步（读者动作、文体锁定、澄清问题）的输入。

## 四、何时需要联网搜索

场景分析**以既有专业知识为主**，多数情况不需要联网。以下情形可辅以 web 搜索补充业界做法：

- 文体较新或冷门（如某垂直行业的合规披露模板），对标杆做法把握不足。
- 需要具体业界数据/标杆来支撑文体选择或篇幅判断。
- 用户提到的做法明显偏离常识，需核实是否已有成熟替代。

搜索聚焦"业界通行文体 / 标杆模板 / 已知陷阱"，不做开放式调研；**基本分析**即可，不展开成深度研究报告。

## 五、专业文档类型选用决策树

按"沟通本质 + 读者动作"定位最匹配的文档类型。**取最具体匹配的；混合场景可组合两类**（如"复盘 + 后续行动计划"）。

```
读者读完要【决策/拍板】？
├─ 重大方案/技术选型需评审            → RFC / 设计文档（Design Doc）
├─ 要资源（预算/人力/立项）           → 立项书 / 商业论证（Business Case）
└─ 记录一个架构/技术决策的"为什么"    → ADR（架构决策记录）

读者读完要【执行/操作】？
├─ 日常运维/应急操作指引             → Runbook / SOP
├─ 接入/使用某个系统或 API           → 操作手册 / 接入指南
└─ 按规范做事                        → 规范 / Standard / Checklist

读者读完要【知晓/同步】？
├─ 周期性进展同步                    → 周报 / 月报 / Status Update
├─ 一次性发布/变更通知               → 变更公告 / Release Notes
└─ 事件/事故对外或对内说明           → 事故公告 / 通报

读者读完要【复盘/学习】？
├─ 事故/项目事后总结                 → 复盘报告（Postmortem）
├─ 沉淀经验/方法论                   → 技术文章 / 最佳实践 / Wiki
└─ 新人上手                          → 入职/上手指南（Onboarding）

读者读完要【约定/约束】？
├─ 双方达成共识的条款                → 协议 / 备忘录 / 会议决议
└─ 团队必须遵守的规则                → 制度 / 规范

读者读完要【定义"做什么"】？（产品视角）
└─ 产品功能定义                      → PRD（产品需求文档）
```

> 定位不清时，先问"读者读完要做什么"（决策/执行/知晓/复盘/约定/定义），再落到文体。**动作比主题更能决定文体。**

## 六、每类文档"专业版必备信息点"（盲区诊断清单）

用于 Checklist 第 2 步诊断盲区：把这些必备点对照用户表述，漏了的即盲区，呈现给用户取舍。（完整结构模板见 `doc-blueprint/references/doc-type-library/`。）

| 文档类型 | 专业版必备信息点（漏了就是盲区） | 典型读者动作 |
|---------|--------------------------------|--------------|
| **RFC / 设计文档** | 背景与问题、现状与约束、**提案**、**备选方案与权衡**、决策、影响/风险、开放问题 | 评审通过/否决方案 |
| **PRD** | 问题与用户价值、目标、**可度量成功指标**、功能范围、**明确不做**、依赖与风险 | 确认开发范围 |
| **ADR** | 背景、**决策**、**理由**、后果/影响、相关 ADR | 记录并认可决策 |
| **立项书 / 商业论证** | 机会/问题、方案、**成本**、**收益(量化)**、风险、里程碑、资源诉求 | 批准资源/预算 |
| **Runbook / SOP** | 触发条件、前置检查、**分步操作(可复制命令)**、预期结果、**回滚步骤**、升级路径 | 按步骤正确执行 |
| **复盘报告** | 摘要、影响、**时间线**、**根因**、**Action(Owner+DDL)**、经验沉淀 | 认领改进项并执行 |
| **周报 / Status** | 本期完成、**风险/阻塞**、下期计划、需协调 | 知晓进展/提供帮助 |
| **变更公告** | 变更内容、**影响范围**、**时间窗口**、**对用户的影响与应对**、联系人 | 知晓并做准备 |
| **事故公告** | 发生了什么、**影响**、**当前状态**、**补偿/应对**、后续跟进 | 安心/采取应对 |
| **会议决议** | 议题、**结论(决策)**、**Action(Owner+DDL)**、遗留问题 | 按结论与 Action 执行 |
| **技术文章 / Wiki** | 问题/目标、方案、**可复现步骤/代码**、踩坑、结论 | 学会并复用 |

> **盲区诊断法**：用户表述里缺了对应行的某个必备点，就在第 2 步呈现给用户——"这类文档通常还要有 X，你需要吗？"

## 七、流程示例：对"写个上个月的事故总结"做场景分析

**定位**：事故总结 → 沟通本质【复盘】→ 读者动作【认领改进项并执行】→ 文体 **复盘报告（Postmortem）**。

【沟通本质】本质是事后学习，关键动作是让相关人认领改进项。不是追责、不是对外公关。
【专业文体】业界通行写 Postmortem，专业版必备：摘要、影响、**时间线**、**根因**、**Action(Owner+DDL)**、经验沉淀。
【盲区】你的表述暂未提到：根因分析（是技术故障还是流程问题？）、改进项（谁、什么时候做、怎么验收）。这些是复盘的核心，通常不能省。
【风险】这类文档常踩的坑：① 只罗列"发生了什么"不说"为什么"和"所以呢"；② Action 没有 Owner 和 DDL，开完全议就忘；③ 变成追责大会，没人敢说真因。

→ 把以上呈现给用户："这些必备点（根因 / Action / Owner / DDL）你要写哪些？另外这是对内复盘还是要对外公告？两者文体差别很大。" 以此驱动澄清。
