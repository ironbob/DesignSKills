# 澄清问题与边界控制

> 配合 clarify-doc skill 的 Checklist 第 3–5 步使用。
> 核心纪律：**一次只问一个问题；多选优先；只问"要达成什么"，不问"正文怎么写"。**
> 本 skill 独立，只负责写作意图澄清，不涉及正文写作 / 结构 / 表现力 / 渲染。

---

## 一、澄清问题分类

按以下 6 类组织提问。先问"读者动作"和"文体"，再逐步收敛到"范围边界"。不要跳着问。

### 1. 读者动作类（最先问）
- 这篇文档主要给谁看？他读完要做什么（决策 / 执行 / 知晓 / 说服）？
- 如果读者读完什么都不用做，这篇文档还存在吗？
- 怎么判断这篇文档"起作用了"？——能给一个可观察的动作或时间点吗？

### 2. 文体类
- 我判断这种情况适合写 <文档类型>，你认同吗？还是要 <备选类型>？
- 这是对内的还是对外的？同一件事对内复盘和对外公告差别很大。
- 需要正式定稿归档，还是内部快速同步？

### 3. 信息点类
- 你希望读者最记住的 1-3 个点是什么？
- 我列了这类文档通常必备的 A/B/C，你看哪些其实不写、哪些必须写？
- 有没有必须包含的硬事实（数字/日期/条款/命令），现在就能给齐吗？

### 4. 范围边界类（靠后问）
- 哪些是你想过但决定先不写的？
- 有没有看起来相关、但其实不该塞进来（否则读者抓不住重点）？
- 本篇最少要传达什么，你才会觉得"够了"？

### 5. 篇幅 / 语气类
- 大概多长？一屏内、一两页、还是详尽长文？
- 语气偏正式严谨，还是内部口语直接？
- 读者是专家还是外行？专业术语能直接用还是要解释？

### 6. 渠道类
- 这篇最终发在哪（Confluence / 企微 / 邮件 / 印刷）？这会影响篇幅与格式约束。
- 有没有硬性格式要求（如必须能贴进企微、需要可复制命令块）？

## 二、好问题 vs 坏问题对照

| ❌ 坏问题（含糊 / 开放式过载） | ✅ 好问题（具体 / 聚焦 / 易答） |
|---|---|
| 你想写什么？ | 这篇主要给谁看，他读完要做什么？ |
| 内容都清楚了吗？ | "读者读完要做什么"——你能给一个可观察的动作吗（比如周五前批准预算）？ |
| 有什么特殊要求？ | 这是对内复盘还是对外公告？两者文体差别很大。 |
| 要写多详细？ | 你希望读者最记住的 1-3 个点是什么？ |
| 你觉得还需要写什么？ | 我列了这类文档通常必备的 A/B/C，哪个其实不写？ |

## 三、越界拉回话术

当用户（或自己的思路）滑向正文 / 结构 / 表现力 / 渲染层时，用以下话术拉回，并记入"未决问题"。**不在本阶段展开这些话题。**

### 正文措辞类（怎么遣词、怎么开头）
> "『这一段怎么写 / 开头怎么起 / 这句怎么表达更专业』是正文写作的事，超出写作意图澄清的范围。我们只定『要传达什么、给谁、要他做什么』。先记一笔到未决问题，留给后续的 `doc-blueprint` 写正文环节。"

### 结构类（章节怎么分、信息怎么排序）
> "『分几节 / 每节标题 / 先说哪个』是文档结构的事，超出范围。这阶段只定信息点。先记一笔，留给 `doc-blueprint`。"

### 表现力类（用图还是表、用什么图）
> "『这里放图还是表 / 用什么图表』是表现力选型，超出范围。先定内容，表现力由 `doc-blueprint` 按专业判断选。"

### 渲染格式类（Markdown / Confluence / 用什么宏）
> "『发到 Confluence 要什么宏 / Markdown 怎么排版』是渲染格式，超出范围。先记一笔，留给 `doc-render`。"

### 通用拉回句式
- "这个问题很好，但它属于 [正文 / 结构 / 表现力 / 渲染] 决策，不在写作意图澄清的范围。"
- "我先记到未决问题。本 skill 到产出写作需求 brief 即结束。"

## 四、范围方案切分模板

当信息点较多、篇幅可伸缩时，提出 2-3 档范围方案，**先说推荐项及理由**。

### 切分维度（按优先级 P0/P1/P2 归档）
- **精简版（只含 P0）**：判据——"读者只看 P0 就能完成期望动作"。适合内部快速同步 / 篇幅受限。
- **标准版（P0 + P1）**：判据——"专业版的完整度，且 P1 里某项直接支撑读者动作"。默认推荐。
- **详尽版（+ P2）**：判据——"归档/培训用途，需要背景和延伸"。适合长期留存的文档。

### 呈现格式
```
建议三档范围：
- 【推荐】标准版（P0 + P1）：含 <关键信息点>。
  理由：专业版完整度够，且 P1 里 <某项> 直接决定读者能否 <期望动作>。
- 精简版（仅 P0）：含 <…>。适合想最快同步、篇幅受限。
- 详尽版（+ P2）：额外 <…>。适合要归档/培训、长期留存。
```

### 切分时的判断要点
- 默认推荐"标准版"，除非用户明确要快速同步（推精简版）或要归档/培训（推详尽版）。
- 主动建议砍 P2（YAGNI）——明确告诉用户"这几个我觉得可以先不写"。
- 不替用户决定 P0/P1 边界，但可以质疑："你把 X 放 P0，但读者动作其实不依赖它，降到 P1 行吗？"
