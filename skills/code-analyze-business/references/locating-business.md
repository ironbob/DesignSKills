# 定位业务 + 建代码地图

> 配合 `code-analyze-business` 的 Checklist 第 2 步使用。这一步**只读不深挖**：目标是把"业务范围"圈出来、把入口和关键文件定位到 `file:line`，呈现给用户确认。深度追踪留给第 4 步。

## 一、为什么不能照名猜业务

类名、路由名、表名是"命名者的视角"，不是"业务真相"：

- `OrderService` 可能只做订单**查询**，下单主流程其实在 `CheckoutService`。
- `/api/refund` 路由可能只是个转发壳，真正的退款决策在某个领域事件处理器里。
- `user_status` 字段名看不出"0=未激活 1=正常"——状态语义要追赋值点。

**正确定位法**：从**触发点**（什么动作启动了这项业务）反追，而非从名字正猜。找到触发点 → 顺着调用链看它实际改了什么，那才是这项业务的边界。

## 二、按入口形态分诊

业务由什么触发，决定从哪里开始定位。先用关键词在代码库搜触发信号，缩小范围：

| 入口形态 | 常见触发信号（grep 关键词示例） | 定位手法 |
|---|---|---|
| HTTP 接口 | `@app.` `@router.` `@Controller` `@GetMapping` `router.post` `@api_view` | 找路由表/控制器，按 URL 路径或业务词匹配 |
| 消息队列 | `@RabbitListener` `@KafkaListener` `consume` `on_message` `@EventHandler` | 找 Topic/Queue 订阅注册，按业务词匹配消息类型 |
| 定时任务 | `@Scheduled` `celery beat` `cron` `schedule` `APScheduler` | 找调度配置表，按业务词匹配 job 名 |
| 事件驱动 | `emit` `publish` `dispatch` `@EventListener` `on(` | 找事件发布点 → 追订阅者 |
| 手动 / CLI | `def main(` `if __name__` `argparse` `click.command` | 找脚本入口 |

> 前后端都要查：C 端一项业务常是"前端页面/组件 → 调接口 → 后端 handler"的链路，入口可能在前端（用户操作）也可能在后端（接口）。

## 三、建代码地图

定位到入口后，沿调用链列出**与这项业务直接相关**的文件，填成一张地图（不展开实现细节，那是第 4 步的事）：

```markdown
### 代码地图：<业务名>

| 角色 | 文件 | 入口/关键位置 | 职责一句话 |
|------|------|---------------|-----------|
| HTTP 入口 | src/api/refund.py | POST /refund : refund_handler() @ src/api/refund.py:42 | 接收退款请求、参数校验 |
| 领域服务 | src/service/refund_service.py | RefundService.refund() @ src/service/refund_service.py:88 | 退款决策、金额计算 |
| 数据访问 | src/repo/order_repo.py | update_status() @ src/repo/order_repo.py:120 | 改订单状态 |
| 外部依赖 | src/clients/payment_client.py | call_refund() @ src/clients/payment_client.py:55 | 调用第三方支付退款 |
```

只收"直接相关"的文件，不要把整条 import 链或工具函数都列进来——地图要聚焦到这项业务。

## 四、识别前后端颗粒度

"一项业务"在不同端颗粒度不同，定位时要分清边界，标出**跨边界点**（前后端衔接处）：

- **前端**：页面 / 组件 → 用户操作 → 调接口。颗粒度通常是"一个用户能完成的操作流"。
- **后端**：handler → service → repository / 外部服务。颗粒度通常是"一个领域能力"。
- **跨边界点**：前端调后端的那个接口，是两边的衔接口——两边的分析都以它为锚。

若业务横跨前后端，地图里要同时出现两端文件，并用"跨边界点"标出衔接接口。

## 五、呈现上下文与业务范围，等用户确认（两段式门禁）

> 第 1 步已向用户索取业务 / 行业文档（若有）。这一步顺手读仓库信号拟「应用画像 + 整体架构 + 领域定位」；**行业惯例锚点按获取链取**（用户文档 > 模型推断）—— 详见 `domain-profiling.md §三`。把上下文与业务范围一起呈现确认，再深挖。

把定位结果用**固定模板**呈现，请用户确认上下文与范围对不对：

```
我理解的【<业务名>】上下文与范围：
- 应用画像：<app 是什么、面向谁>
- 整体架构：<技术形态与边界，一句话>
- 领域定位：<本业务在 app/领域里的位置>
- 行业惯例锚点：<...>
- 领域知识来源：user-doc / model-inferred（模型推断的逐条待你确认）
- 触发词：<用户操作 / 接口 / 事件 / 定时>
- 入口：<文件:行号 + 入口标识>
- 边界：包含 <A、B、C>，不包含 <D、E>
- 代码地图：（上方表格）

这个 app 全貌、领域定位、行业惯例与业务范围对吗？要修正/增减哪部分？
```

**未确认前不进第 4 步。** 用户说"范围不对"或"app 画像不对"就回来重新定位（可能入口找错了，或边界划宽/划窄了，或领域判断偏了）。这一步拦住最大的失败：分析了一项根本不是用户想问的业务，或在错误的领域 framing 下反推需求。
