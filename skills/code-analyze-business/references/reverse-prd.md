# 反推需求：把现状翻译成 PM 视角

> 配合 `code-analyze-business` 的 Checklist 第 8 步使用。前提：`analysis.md` 已通过 `validate_analysis`。这一步把 analysis 里的**实现行为**，翻译成产品经理能读的**需求文档**（`requirements.md`）。结构与 `clarify-requirements` 的正向需求对齐，但有反推专属的两处增量。

## 一、反推 vs 正向（先想清楚这步在干什么）

| | 正向需求（clarify-requirements） | 反推需求（本步骤） |
|---|---|---|
| 输入 | 模糊想法 | 已实现代码（analysis） |
| 产出 | **要**做什么功能 | 这业务**已满足**的需求是什么 |
| 独有产物 | 无 | ① 功能清单「实现状态」列 ② 第 7 节「实现与需求偏差」 |

反推多出的两样，正是正向 PRD 给不了的——也是反推最有价值的部分：你能看出"代码做了但需求未必要的"和"需求该有但代码没做的"。

## 二、逐节翻译规则（analysis 节 → requirements 节）

| requirements 节 | 来源 analysis 节 | 翻译手法 |
|---|---|---|
| 1. 一句话目标 | §1 业务概述 | 把"这项业务是什么"翻成"为**谁**解决**什么问题**"（加主语和痛点） |
| 2. 目标用户 | §2 触发与入口 | 谁会触发、谁受益；**NOT 目标** = 代码明确没服务的人 |
| 3. 核心场景 | §4 主流程 + §5 分支 | 每个主路径/分支 → `触发情境 → 用户目标 → 期望结果` |
| 4. 功能清单 ★ | §4/§5/§6 | 把代码能力拆成功能；优先级 + 实现状态 + 回链（见下两节） |
| 5. 明确不做 | 代码没实现的 | 如"不支持一笔订单分多次部分退" |
| 6. 非功能约束 | §7 依赖 + §8 风险 + 完整性 5 项 | 超时/并发/幂等/外部 SLA 翻成产品约束 |
| 7. 实现与需求偏差 ★ | （反推独有） | 对比"代码做的" vs "需求应做的"，见第四节 |
| 8. 已知缺口 | analysis 已知缺口 | 对齐引用，互相指向 |

## 三、实现状态怎么判（功能清单的核心列）

每条功能标三态，依据是能否在 analysis 里找到完整回链：

| 状态 | 含义 | 判定 |
|---|---|---|
| ✅ 已实现 | 代码完整支撑该验收标准 | analysis 有明确回链、无 ⚠ 未确认 |
| ⚠️ 部分 | 实现了主体但规则不全/有未确认点 | 回链存在，但某条规则标 ⚠ 未确认 |
| ❌ 缺口 | 需求层面应有，代码没做 | 推断应有（见第五节），计入 `gaps` |

> `gaps` 计数 = ❌ 缺口数。validator 会核它和正文 ❌ 状态、⚠ 未确认 对得上。

## 四、优先级反推（不是 PM 排的，是从代码结构反推）

- **P0** = 主流程必经路径（analysis §4 happy path 上的每一步）。
- **P1** = 主要分支 / 异常路径（analysis §5）。
- **P2** = 边缘 / 可选能力。

> 这是"实现反映出的重要度"，和正向 PRD 由 PM 排优先级不同——反推时如实标注，若你判断与产品本意不符，记到第 7/8 节。

## 五、偏差分析（第 7 节，反推独有价值）

对比"代码实际做的"与"需求层面应做的"，分两类：

1. **过度实现 / 副作用**（代码做了，需求未必需要）
   - 问："这是需求要的，还是实现的副作用？"
   - 例：退款失败只入补偿队列、不向用户报错（代码行为）→ 需求层面是否该通知用户？⚠ 未确认。
2. **需求缺口**（需求该有，代码没做）
   - 从业务常识 / 行业惯例推断"通常该有 X"，但代码里没找到。
   - 例：补偿 job 的重试上限/告警不在本业务内 → 推断"通常该有上限与告警" ⚠ 未确认。

> 推断的"应有需求"一律标 ⚠ 未确认，进第 8 节，不混进功能清单当既成事实。

## 六、验收标准的颗粒度（解决"细节不够细"）

把 analysis 的实现细节提炼成**可判定的断言**，而非空话：

| ❌ 泛述 | ✅ 可判定（回链 analysis） |
|---|---|
| 做了可退校验 | 仅状态∈{已支付(1)}且金额≤可退余额的订单可退 ……`refund_service.py:95` |
| 调了支付系统 | 调用 `PaymentClient.call_refund`，超时 3s、失败重试 2 次 ……`payment_client.py:55` |
| 做了并发控制 | Redis 锁 key=order_id 串行化 + refund_records.order_id 唯一索引 ……`refund_service.py:160` |

判定标准：能点名具体函数 / 阈值 / 枚举值；禁 banned 词（`体验好/很重要/适当处理…`）。

## 七、反推防臆造（沿用 analysis 的底线）

- 每条功能 / 验收标准都回链 analysis 章节 或 `file:line`。
- 推断的"应有需求"标 ⚠ 未确认，登记第 8 节，不进正文当事实。
- 不臆造 analysis 里没有的功能；功能清单是"代码已实现的"＋"推断应有但缺失的（标 ❌）"。

## 八、与正向需求的边界

- **只反推已实现的需求**，不写"新产品该做什么"（那是 `clarify-requirements`）。
- 不提"该重构成什么""体验怎么优化"（那是后续 skill）。
- 当对话滑向"这块该加个功能""应该重构成 XX"，明确"这超出反推需求范围，只记现状与缺口"，记一笔到第 8 节。
