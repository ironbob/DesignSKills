# 测试用例生成：覆盖规则 + 边缘 case 推导

> 配合 `code-analyze-business` 的 Checklist 第 9 步使用。前提：`analysis.md` + `requirements.md` 均已通过校验。这一步从需求 + 现状派生**覆盖所有需求与边缘 case 的测试用例**（`test-cases.md`）。

## 一、覆盖规则（硬性，validator 会查）

- **P0 功能** → ≥1 Happy + ≥1 Error + ≥1 Edge
- **P1 功能** → ≥1 Happy + ≥1 Edge
- **P2 功能** → ≥1 用例（任一类型）
- **analysis 完整性 5 项**（异常分支 / 触发条件 / 并发时序 / 外部依赖 / 幂等）每项至少 1 个用例

> 先把 requirements §4 功能清单逐条过，给每条功能补够用例；再过完整性 5 项，补漏的 Edge Case。

## 二、三 Type 是什么

| Type | 定义 | 典型来源 |
|---|---|---|
| Happy Path | 主流程正常走通 | analysis §4 每个主路径步骤 |
| Error Flow | 业务规则拦截 / 预期失败路径 | analysis §5 分支与异常（校验失败、状态非法） |
| Edge Case | 边界 / 极端 / 并发 / 外部异常 | 完整性 5 项 + 下方边缘清单 |

## 三、完整性 5 项 → 用例映射

analysis 完整性自检的每一项，都要能指到一个至少一个测试用例：

| 完整性项 | 主要用例类型 | 怎么测 |
|---|---|---|
| 异常分支 | Error Flow | 走每个 try/catch、每个错误码返回路径 |
| 触发条件 | Error Flow / Edge | 破坏前置条件（如订单非"已支付"） |
| 并发时序 | Edge Case | 并发发起同一请求 |
| 外部依赖 | Edge Case | 依赖超时 / 返回失败 |
| 幂等 | Edge Case | 相同标识重复提交 / 重放 |

## 四、边缘 case 推导清单（逐项问"有没有"，别漏）

对每个功能，按这张清单过一遍，能想到的就写成 Edge Case：

| 类别 | 该问的 |
|---|---|
| 空值 / 缺失 | 必填参数缺失（无 amount）、空字符串、null |
| 越界 | 金额 > 可退余额、负数、0、超大数 |
| 超时 | 外部依赖超过阈值（>3s） |
| 重试 | 失败重试到上限（重试 2 次仍失败） |
| 并发 | 同一资源并发请求（并发双退） |
| 幂等重放 | 相同标识再次提交 |
| 状态机非法迁移 | 从非法/终态触发（已退款→再退、未支付→退款） |
| 外部依赖失败 | 返回失败码、连接拒绝 |
| 边界值 | 刚好等于阈值（金额=余额、最大长度） |

## 五、写好用例的要点

**Steps（语言无关，从行为角度写）**：
- 从"调了什么接口 / 改了什么状态"角度写，**不写 UI 点击**（这是与 `web-test-case-man` 的关键区别——它绑 FastAPI+Vue，我们语言无关）。
- 具体参数：`order_id=100, amount=80`，不写"一个订单退一点钱"。
- 一条 Step 一个动作，编号清晰。

**Expected Result（可观察、可断言、必回链）**：
- 写成能断言的结果：状态值、返回码、是否建单、是否调第三方。
- **必须回链 `file:line`**（锚到 analysis 的真实代码行为），使它既是用例又是行为规约。
- 负向用例须说明"什么没发生"（如"不调第三方、不建退款单"）。

**Preconditions**：只写开始前必须为真的状态，不写流程本身。

**需求来源**：每个用例标 `REQ-<MODULE>-<n>` 回链 requirements §4 功能清单项，保证"需求→用例"可追溯。

## 六、与 web-test-case-man 的区别（去栈绑定）

| | web-test-case-man | 本 skill |
|---|---|---|
| 输入 | requirements + **tech-design** | analysis + requirements（**不要** tech-design） |
| 产出 | 测试用例文档 **+ Playwright 代码** | 仅测试用例文档（语言无关） |
| 栈 | 绑死 FastAPI + Vue | 语言无关 |
| Expected Result | UI/DOM 断言 | 行为规约 + `file:line` 回链 |
| Steps | 用户视角（点击/输入） | 实现视角（调接口/改状态） |

> 本 skill 不产出可执行测试代码——那是栈绑定的下游 skill（如 web-test-case-man）的事。这里产出的是**语言无关的行为规约型用例**。
