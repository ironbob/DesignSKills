# 应用画像 + 领域定位：先知道这个 app 是干什么的

> 配合 `code-analyze-business` 的 Checklist 第 1→3 步使用。这一步在读代码之前/之中，先搞清两件事：① 这个 app 是什么、整体架构、本业务在领域里的位置（**项目级**，从仓库信号推断）；② 这个领域的**通用知识 / 行业惯例**（**领域级**，按两级获取链取）。两者合写进「应用与领域定位」，并入两段式确认让用户拍板。
>
> **为什么需要**：脱离 app 全貌只盯一个功能，反推出来的"需求"会贴着代码功能列，而不是从专业领域角度讲业务规则。先有领域 framing，requirements 的验收标准才写得像 PM、test-cases 的断言才写得像黑盒测试。

---

## 一、读哪些信号推断"这个 app 是什么"（项目级，只读不深挖）

从仓库级、非业务深度的信号里拼出 app 全貌（app 画像 / 架构 / 领域定位）：

| 信号 | 位置 | 能看出什么 |
|------|------|-----------|
| 包描述 / manifest | `package.json:description`、`go.mod`、`pom.xml`、`pyproject.toml`、`requirements.txt` | app 名、用途、技术栈、关键依赖（支付/IM/存储 SDK 暴露领域） |
| README / 文档 | `README.md`、`docs/`、`CHANGELOG` | 产品定位、面向谁、核心能力自述 |
| 顶层目录结构 | 顶层一级目录 | 架构形态（monorepo？前后端分离？微服务？模块化？）、业务边界 |
| 入口与路由总表 | `app.py` / `main` / `routes` / `router` 注册处、前端 `pages|views` 列表 | app 暴露了哪些领域能力、规模 |
| 数据模型 / migration | `model/`、`entity/`、`migrations/`、`schema.sql` | 核心实体、领域名词、状态字段 |
| 配置 / 环境变量 | `.env.example`、`config`、`application.yml` | 接了哪些外部系统（支付通道、短信、对象存储 → 暴露领域） |

> 这些是"猜 app 是什么"的线索，不是要全读；抓 3–5 条足够拼出画像。**找不到的标 `⚠ 未确认`，别编。**
> 注意：这一节推断的是**项目自身信息**（这个 app 是什么）。**行业通行规则**这类通用知识读不出来，见 §三 获取链。

## 二、拟「应用与领域定位」（4 件套）

把线索浓缩成四句话，写进 analysis 的 `## 应用与领域定位` preamble 与 frontmatter：

1. **应用画像 `app_profile`** —— 这个 app 是什么、面向谁、解决什么（一句话）。例："面向 C 端用户的电商订单中台，负责下单/支付/履约/售后的订单生命周期。"
2. **整体架构 `architecture`** —— 技术形态与边界（一句话）。例："Vue 前端 + Python 单体后端 + 第三方支付/物流，Redis 做缓存与分布式锁，MQ 做异步解耦。"
3. **领域定位** —— 本业务在 app/领域里的位置。例："退款是订单生命周期「售后」段的终态能力，前置依赖支付成功，后接触资损结算。"
4. **行业惯例锚点 `domain_anchor`** —— 该领域的通行规则 / 合规底线，用于反推时判断"通常该有什么"。**它的来源走 §三 获取链**，并在 frontmatter `domain_source` 记录主源。

## 三、领域通用知识（行业惯例锚点）的获取链 ★

行业通行规则 / 合规底线这类**通用领域知识**，仓库技术信号读不出来。**按以下优先级获取**，高优先级覆盖低优先级，并把主源记进 frontmatter `domain_source`：

| 优先级 | 来源 | 怎么取 | `domain_source` |
|--------|------|--------|-----------------|
| ① 最高 | **用户提供的业务文档** | **开始分析前主动问用户**（见下"前置动作"）。有则读，作为主源。 | `user-doc` |
| ② 兜底 | **模型推断** | 用户没给业务文档时，用模型行业知识推断 —— **必须输出领域知识块待用户确认**（见下），不许悄悄用。 | `model-inferred` |

### 前置动作（Checklist 第 1→2 步之间，必做）

开始深挖前，先问用户：

> 这个业务有没有现成的**业务 / 行业文档**（PRD、合规要求、风控规则、行业规范、历史需求）？有的话**贴内容或给文件路径**。
- **有** → 读取作为领域知识主源（`domain_source=user-doc`），后续行业惯例以它为准。
- **没有** → 走 ② 模型推断（明牌输出待用户确认）的链。

### 模型兜底必须"明牌输出 + 待确认"（第 ② 级硬要求）

当走到模型推断（用户没给业务文档时）：

1. 把推断的行业惯例**单独成块**写进 analysis「应用与领域定位」节，块头显式标注：`⚠ 未确认 · 模型推断 · 待用户确认`。
2. 每条标依据类型（模型常识 / 类比同类系统），**不伪装成项目事实**。
3. 登记「已知缺口」，在两段式确认 / 交付时请用户**逐条确认或修正**。
4. 用户确认前，这些推断只在偏差分析 / 用例推导里当"疑似尺子"，**不写进 requirements 功能清单当已实现能力**。

> 即：模型兜底 = 明牌推断 + 输出待确认，不是默默填一个 domain_anchor 就完事。

### 来源冲突

高优先级覆盖低优先级：用户文档 > 模型推断。冲突以用户文档为准，并在「已知缺口」记一笔差异（如"模型推断通常有 X，但用户文档说本项目不做 X"）。

## 四、领域知识如何下游化用

定位不是写完就完，它要**贯穿后两件套**：

| 下游产出 | 怎么用领域知识 |
|---------|---------------|
| requirements 验收标准 | 用领域 / 用户语言写断言（"已支付订单可退""金额 ≤ 可退余额"），不写代码符号 —— 见 `reverse-prd.md` |
| requirements 偏差分析 | 用行业惯例锚点判断"通常该有 X 而代码没做" → 需求缺口 |
| requirements 明确不做 | 用领域边界佐证"本 app 不服务谁 / 不做哪个能力" |
| test-cases 断言与命名 | 用领域概念命名用例、用用户可观察结果写 Expected Result —— 见 `test-case-generation.md` |
| analysis 领域概念节 | 术语表用领域专业词（"退款单 / 可退余额"），不照搬代码字段名 |

## 五、防臆造（沿用 analysis 底线）

- 应用画像 / 架构 / 领域定位能从仓库信号证实就写；**信号不足标 `⚠ 未确认`**，不脑补。
- 行业惯例锚点按 §三 获取链取，**来源必须诚实标注**（`domain_source`）；模型推断的标 `⚠ 未确认` + 明牌成块 + 登记缺口。
- **绝不把"行业通常有 X" / "模型推断 X"当成"本项目实现了 X"写进功能清单。**
- 画像 / 锚点只作 framing 与下游尺子，不替代码事实 —— 实现行为仍以 analysis 回链为准。

## 六、呈现给用户确认（并入两段式确认模板）

在 `locating-business.md` 第五节的范围确认模板里，追加领域一项，并标出来源：

```
我理解的【<业务名>】上下文：
- 应用画像：<app_profile>
- 整体架构：<architecture>
- 领域定位：<本业务在 app 里的位置>
- 行业惯例锚点：<domain_anchor>
- 领域知识来源：user-doc / model-inferred（模型推断的逐条待你确认）
这个 app 全貌、领域定位与行业惯例对吗？要修正哪句？
```

用户确认（或修正）后才进第 4 步深挖。
