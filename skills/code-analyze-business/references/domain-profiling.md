# 应用画像 + 领域定位：先知道这个 app 是干什么的

> 配合 `code-analyze-business` 的 Checklist 第 2 步使用。这一步在读代码之前/之中，先从**仓库级信号**推断"这个 app 是什么、整体架构、所处领域、行业惯例"，拟一份**应用与领域定位**，并入两段式确认让用户拍板。
>
> **为什么需要**：脱离 app 全貌只盯一个功能，反推出来的"需求"会贴着代码功能列，而不是从专业领域角度讲业务规则。先有领域 framing，requirements 的验收标准才写得像 PM、test-cases 的断言才写得像黑盒测试。

## 一、读哪些信号（只读不深挖）

从仓库级、非业务深度的信号里拼出 app 全貌：

| 信号 | 位置 | 能看出什么 |
|------|------|-----------|
| 包描述 / manifest | `package.json:description`、`go.mod`、`pom.xml`、`pyproject.toml`、`requirements.txt` | app 名、用途、技术栈、关键依赖（支付/IM/存储 SDK 暴露领域） |
| README / 文档 | `README.md`、`docs/`、`CHANGELOG` | 产品定位、面向谁、核心能力自述 |
| 顶层目录结构 | 顶层一级目录 | 架构形态（monorepo？前后端分离？微服务？模块化？）、业务边界 |
| 入口与路由总表 | `app.py` / `main` / `routes` / `router` 注册处、前端 `pages|views` 列表 | app 暴露了哪些领域能力、规模 |
| 数据模型 / migration | `model/`、`entity/`、`migrations/`、`schema.sql` | 核心实体、领域名词、状态字段 |
| 配置 / 环境变量 | `.env.example`、`config`、`application.yml` | 接了哪些外部系统（支付通道、短信、对象存储 → 暴露领域） |

> 这些是"猜 app 是什么"的线索，不是要全读；抓 3–5 条足够拼出画像。**找不到的标 `⚠ 未确认`，别编。**

## 二、拟「应用与领域定位」（4 件套）

把上面线索浓缩成四句话，写进 analysis 的 `## 应用与领域定位` preamble 与 frontmatter：

1. **应用画像 `app_profile`** —— 这个 app 是什么、面向谁、解决什么（一句话）。例："面向 C 端用户的电商订单中台，负责下单/支付/履约/售后的订单生命周期。"
2. **整体架构 `architecture`** —— 技术形态与边界（一句话）。例："Vue 前端 + Python 单体后端 + 第三方支付/物流，Redis 做缓存与分布式锁，MQ 做异步解耦。"
3. **领域定位** —— 本业务在 app/领域里的位置。例："退款是订单生命周期「售后」段的终态能力，前置依赖支付成功，后接触资损结算。"
4. **行业惯例锚点 `domain_anchor`** —— 该领域的通行规则/合规底线，用于反推时判断"通常该有什么"。例："电商退款行业惯例：资金须原路退回、同一笔交易不可超额/重复退款、需可追溯（留退款单）。"

> 行业惯例是**推断的参照系**，不是本项目既成事实 —— 标 `⚠ 未确认`，进已知缺口。它只在"反推需求/偏差分析/用例推导"时当尺子，不混进功能清单当已实现能力。

## 三、领域知识如何下游化用

定位不是写完就完，它要**贯穿后两件套**：

| 下游产出 | 怎么用领域知识 |
|---------|---------------|
| requirements 验收标准 | 用领域/用户语言写断言（"已支付订单可退""金额≤可退余额"），不写代码符号 —— 见 `reverse-prd.md` |
| requirements 偏差分析 | 用行业惯例锚点判断"通常该有 X 而代码没做" → 需求缺口 |
| requirements 明确不做 | 用领域边界佐证"本 app 不服务谁/不做哪个能力" |
| test-cases 断言与命名 | 用领域概念命名用例、用用户可观察结果写 Expected Result —— 见 `test-case-generation.md` |
| analysis 领域概念节 | 术语表用领域专业词（"退款单/可退余额"），不照搬代码字段名 |

## 四、防臆造（沿用 analysis 底线）

- 应用画像/架构能从仓库信号证实就写；**信号不足的标 `⚠ 未确认`**，不脑补。
- 行业惯例锚点是推断参照系，一律 `⚠ 未确认` + 登记已知缺口；**绝不把"行业通常有 X"当成"本项目实现了 X"写进功能清单。**
- 画像只作 framing 与下游尺子，不替代码事实 —— 实现行为仍以 analysis 回链为准。

## 五、何时自己判断，何时问用户

- **默认自拟 + 确认**：先用上面的信号拟出 4 件套，在两段式确认里一并呈现，让用户确认/修正。这与现有 HARD-GATE 一致，不额外加门槛。
- **信号极弱时显式追问**：仓库无 README、manifest 含糊、monorepo 里定位不到子模块归属、入口不明 —— 直接请用户补一句"这个 app 是做什么的 / 整体架构 / 所处领域"，别硬猜。

## 六、呈现给用户确认（并入两段式确认模板）

在 `locating-business.md` 第五节的范围确认模板里，追加领域一项：

```
我理解的【<业务名>】上下文：
- 应用画像：<app_profile>
- 整体架构：<architecture>
- 领域定位：<本业务在 app 里的位置>
- 行业惯例锚点：<domain_anchor>（⚠ 未确认，作反推尺子）
这个 app 全貌与领域定位对吗？要修正哪句？
```

用户确认（或修正）后才进第 4 步深挖。
