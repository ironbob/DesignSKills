# 质量规则：防臆造 + 完整性 + 专业

> 配合 `code-analyze-business` 的 Checklist 第 5、6 步使用。第 5 步用它逐项过完整性清单；第 6 步自审也用它。

## 一、防臆造 3 原则（底线）

1. **回链** —— 每条结论必须能指到 `file:line`。指不到的不写进正文当事实。
2. **推断标未确认** —— 合理推断但没在代码直接证实的，标 `⚠ 未确认`，并在"已知缺口"登记。
3. **图边对回真实调用** —— Mermaid 图每条边都要在下方文字有对应回链；禁止画代码里没有的边。

> 契约与质量解耦：回链格式 / 覆盖率 / banned 词 / 图围栏由 `validate_analysis.py` 机器卡；真实性与深度靠下面的人工自审。

## 二、完整性自检清单（5 项，逐项过，不留空）

对这项业务逐项问"有没有"，每项给出**"有 / 无 / 不适用 + 依据"**，不能空着：

| # | 维度 | 要查什么 | 例 |
|---|------|---------|----|
| 1 | 异常分支 | try/catch、错误码、失败返回路径 | 退款失败是否 catch、返回什么错误码 |
| 2 | 触发条件 | 守卫、参数校验、权限校验 | 仅"已支付"可退、金额上限校验 |
| 3 | 并发时序 | 锁、事务、竞态、乱序 | 重复退款请求是否加锁 / 幂等键 |
| 4 | 外部依赖 | 下游服务失败如何处理 | 支付中心超时是否重试 / 补偿 |
| 5 | 幂等 | 重复触发、去重键 | 同一退款单重复请求是否幂等 |

每项在正文（异常·兜底·兼容 / 依赖与耦合 / 风险与技术债 节）要有结论；脚本会扫这 5 个关键词是否覆盖。

## 三、Banned 词（生成与自审禁用，脚本会扫）

`体验好` `功能完善` `功能强大` `适当处理` `待定` `待补` `后续再说` `良好体验` `非常重要` `很关键` `等等` `TBD` `TODO`

→ 改成**具体可核**的说法（见专业规则）。

改写示例：

| ❌ 空话 | ✅ 具体可核 |
|---|---|
| 退款时做了适当处理 | 失败重试 3 次，仍失败则置"退款中"待人工 …… `refund_service.py:140` |
| 这个功能很重要 | 退款是订单生命周期的终态之一，影响资损结算 |
| 性能很好 | 退款接口 P99 < 200ms ⚠ 未确认（未实测） |

## 四、专业规则（自审逐条过）

> **分层铁律（最关键，先记）**：`analysis`（现状层）的代码术语随便用——代码就是主角；但 `requirements` / `test-cases` 的断言用**业务语言**，代码事实（函数/路由/表名/枚举/状态码）一律归「实现锚点」，不写进断言正文。详见 `reverse-prd.md §六`、`test-case-generation.md §五`。

| # | 规则 | ❌ 套话 | ✅ 专业 |
|---|------|--------|------|
| 1 | **点名具体函数 / 模块**（仅 analysis 层） | "调用支付系统" | "调用 `PaymentClient.call_refund` @ payment_client.py:55" |
| 2 | **因果链，非断言** | "加了锁防并发" | "无锁 → 同一订单并发退款可能双退 → 用 `refund_lock` 串行化 @ :160" |
| 3 | **量化阈值回链** | "响应要快" | "退款接口 P99 < 200ms ⚠ 未确认（未实测）" |
| 4 | **不泛指** | "做了校验" | "校验订单状态 ∈ {已支付} 且金额 ≤ 可退余额" |
| 5 | **对通用模式有增量** | 复述框架默认行为 | 点出本项目特有的处理（如自定义退款规则） |
| 6 | **框架 / 规范素养** | "符合规范" | 事务边界、幂等键、状态机语义点名清楚 |
| 7 | **业务语言 vs 代码术语分层**（requirements / test-cases） | 验收标准："仅状态∈{已支付(1)}…refund_service.py:95" | 验收标准："仅'已支付'订单可退，金额 ≤ 可退余额"；代码符号归「实现锚点」 |
| 8 | **主流程讲透决策与兜底**（analysis） | "计算退款金额 —— 按规则算 …:110" | 每步写做什么 + 适用项的异常/兜底/兼容；系统覆盖见 §5 矩阵 |

## 五、回链有效性自检

- 路径是否真实存在于代码库？（脚本不查，靠人工 / IDE 核对）
- 行号是否对得上结论描述的代码？（抽检 10 条，目标 ≥9/10）
- 跨文件引用是否一致？（如"入口调 refund_service"在地图、主流程、依赖节描述都要对得上）
