# 反推功能需求文档模板

> 本 skill 的产出物之一。从已确认的 `analysis.md`（现状事实源）**反推**出 PM 视角的需求：这项已实现的业务，**为谁解决什么问题、该满足哪些功能、验收标准是什么**。
> 章节结构与 `clarify-requirements` 的正向需求模板对齐，但有**反推专属**的两处增量：① 功能清单的「实现状态」列；② 第 7 节「实现与需求偏差」。
>
> **反推铁律**：反推的是**已实现的需求**，不是未来需求。每条功能/验收标准都要回链到 `analysis.md` 的章节或 `file:line`；代码里没直接证实的「应有需求」标 `⚠ 未确认` 并计入第 8 节，不混进正文当既成事实。

---

## 一、模板

### frontmatter（轻量 YAML）

```yaml
---
business: refund                # 业务名，kebab-case 英文，与同组 analysis 一致
title: 订单退款功能需求（反推）   # 中文标题
summary: 用户对已支付订单发起退款，系统校验后调第三方退款并更新订单状态   # PM 口吻一句话
codebase: order-service          # 代码库（与 analysis 同）
entry_type: http                 # 与 analysis 同：http / mq / scheduled / event / cli / manual
analyzed_at: 2026-06-16          # 与 analysis 同一天
status: confirmed                # draft / confirmed / example
source_analysis: 2026-06-16-order-refund-analysis.md   # ★ 回链事实源（同组 analysis 文件名）
reverse_engineered: true         # ★ 标记这是从代码反推的，非正向需求
gaps: 2                          # ★ 需求层面应有但代码未实现的缺口数（对应 ❌ 状态 + ⚠ 未确认）
---
```

> 本文件不重复贴代码细节（那是 analysis 的事）；它用 PM 语言把 analysis 里的实现行为**翻译成需求**。要喂 `doc-blueprint` 渲染时由 doc-blueprint 声明其单一源。

### 正文章节（7 节 + 偏差节 + 已知缺口）

> 每节给「写什么 / 反推要点」。标 ★ 的是核心或反推独有章节。每条功能 / 验收标准都要回链 analysis 章节或 `file:line`。

#### 1. 一句话目标
<!-- 把 analysis §1「业务概述」翻译成 PM 语言：谁 + 解决什么问题。1-2 句。 -->
- <谁> 用这项业务 <解决什么问题> —— 见 analysis §1 业务概述

#### 2. 目标用户
<!-- 从 analysis §2 触发与入口反推：谁会触发、谁受益。 -->
- 主要用户：<角色 + 关键特征>
- 次要用户：<如有；无则写"无">
- **不是**目标用户：<明确排除，防范围蔓延>

#### 3. 核心场景
<!-- 从 analysis §4 主流程 + §5 分支反推成用户故事：触发情境 → 用户目标 → 期望结果。2-4 个。 -->
- 场景1：<触发情境 → 用户目标 → 期望结果> —— 见 analysis §4
- 场景2：<...>

#### 4. 功能清单 ★
<!-- 按模块分组。这是 PM 视角的核心：把代码能力翻译成"功能 + 可判定验收标准"。两列独有：「实现状态」「回链」。 -->

#### 模块：<模块名>
| REQ id | 优先级 | 功能 | 简述（做什么） | 验收标准（业务语言，可判定） | 实现状态 | 实现锚点 |
|--------|--------|------|----------------|------------------------------|----------|----------|
| REQ-<MODULE>-01 | P0 | <功能名> | <做什么> | <业务语言断言：什么条件下能/不能做什么；禁代码符号与空话> | ✅ 已实现 / ⚠️ 部分 / ❌ 缺口 | analysis §n 和/或 `路径:行号` |
| REQ-<MODULE>-02 | P1 | <功能名> | <做什么> | <...> | ... | ... |

> - **REQ id（join key）**：每条功能给稳定 id `REQ-<MODULE>-<n>`（模块内从 01 递增）。它是 test-cases「需求来源」回链的目标，也是 `<biz>-requirements.json` 的主键——没有它,test-cases 的引用会悬空、覆盖也无从按功能校验。
> - **优先级反推**：P0 = 主流程必经路径（analysis §4 happy path）；P1 = 主要分支/异常（analysis §5）；P2 = 边缘/可选。
> - **实现状态**（反推独有）：✅ 已实现 = 代码完整支撑；⚠️ 部分 = 代码实现了但规则不全/有未确认；❌ 缺口 = 需求层面应有但代码没做（计入 §8 + `gaps`）。
> - **验收标准 = 业务可观测断言**（核心）：用领域/用户语言，**不写**函数名/路由/表名/枚举值/状态码。例：✅ "仅'已支付'订单可退，退款金额 ≤ 可退余额"；❌ "仅状态∈{已支付(1)}…返回 400 + `OrderNotRefundableError`"。代码事实归「实现锚点」列。详见 `reverse-prd.md §六`。

> 除本 md 外，**同步产出 `<biz>-requirements.json`**（机器校验的契约源），把 §4 功能清单结构化。md §4 与 json 必须一致（`validate_requirements_json.py` 会做行数/id/状态/gaps 对账）。

#### requirements.json 结构
```json
{
  "business": "<biz>",
  "source_analysis": "<同组 analysis>.md",
  "gaps": 0,
  "features": [
    {"id": "REQ-<MODULE>-01", "module": "<模块>", "name": "<功能名>",
     "priority": "P0|P1|P2",
     "status": "implemented|partial|gap",
     "anchor": "<路径:行号>"}
  ]
}
```
> `status` 枚举 ↔ md 符号：`implemented`↔✅、`partial`↔⚠️、`gap`↔❌。`gaps` 须 == `status==gap` 的条数。校验：`validate_requirements_json.py <req.json> [<req.md>]`。

#### 5. 明确不做（Out of scope）
<!-- 从代码明确没实现的能力反推边界。每条带原因。 -->
- 不做 <X>：<代码现状/原因，如"退款仅单次全额或单次部分，不支持一笔订单分多次部分退" —— analysis §n>

#### 6. 非功能约束
<!-- 从 analysis §7 依赖耦合 + §8 风险 + 完整性 5 项反推为产品约束。 -->
- 性能/时效：<如"退款接口超时 3s"> —— analysis §7
- 可靠性/并发：<如"同一订单并发退款不双退（幂等）"> —— analysis §8
- 外部依赖：<如"依赖支付中心，失败重试 2 次"> —— analysis §7

#### 7. 实现与需求偏差 ★（反推独有，核心增值）
<!-- 正向 PRD 写不出这一节：它对比"代码实际做的"与"需求层面应做的"，分两类。这是从代码反推需求时最有价值的产物。 -->
- **过度实现 / 副作用**（代码做了，需求未必需要）：<现象> → <为何可能多余/有风险> —— analysis §n
- **需求缺口**（需求层面应有，代码未实现）：<应有能力> → <为何推断应有 + 当前代码现状> ⚠ 未确认 —— analysis §n

#### 8. 未决问题 / 已知缺口
<!-- 对齐 analysis「已知缺口」，互相引用。每处 ⚠ 未确认 都在这里有对应条目。 -->
- <问题>：<为何未确认 / 后续如何证实> —— 见 analysis 已知缺口

## 二、端到端示例

> 示例：订单服务的「退款」业务反推需求（与同组 `analysis.md` 配对，**代码路径为虚构示例**）。

```yaml
---
business: order-refund
title: 订单退款功能需求（反推）
summary: 用户对已支付订单发起退款，系统校验可退条件后调用第三方支付退款，并更新订单状态为已退款
codebase: order-service
entry_type: http
analyzed_at: 2026-06-16
status: example
source_analysis: 2026-06-16-order-refund-analysis.md
reverse_engineered: true
gaps: 2
---
```

### 1. 一句话目标

让已付款但需要撤销交易的用户，能自助发起退款，系统自动校验可退条件并完成资金退还。 —— 见 analysis §1 业务概述

### 2. 目标用户

- 主要用户：已支付订单的 C 端用户（想取消已付款交易）。
- 次要用户：处理退款异常的运营/客服（介入补偿队列里的失败退款）。
- **不是**目标用户：未支付订单的用户（无款可退）、商家后台（退款发起方是用户而非商家）。

### 3. 核心场景

- 正常退款：用户在"订单详情 → 申请退款"点击 → 系统校验订单可退 → 调第三方退款 → 订单变"已退款"。 —— analysis §4
- 不可退拦截：用户对非"已支付"订单点退款 → 系统拒绝，提示原因。 —— analysis §5
- 退款失败兜底：第三方退款失败 → 系统记录失败并入补偿队列，不直接报错给用户。 —— analysis §5

### 4. 功能清单

#### 模块：退款发起
| REQ id | 优先级 | 功能 | 简述 | 验收标准（业务语言，可判定） | 实现状态 | 实现锚点 |
|--------|--------|------|------|------------------------------|----------|----------|
| REQ-REFUND-01 | P0 | 发起退款 | 用户对已支付订单提交退款申请 | 用户可对已支付订单提交退款申请，系统接收并处理 | ✅ 已实现 | analysis §2 · `src/api/refund.py:42` |
| REQ-REFUND-02 | P0 | 可退校验 | 校验订单是否满足退款条件 | 仅"已支付"的订单可申请退款；其他状态被拒绝并提示"不可退" | ✅ 已实现 | analysis §5 · `src/service/refund_service.py:95` |
| REQ-REFUND-03 | P0 | 计算退款金额 | 按规则算全额或部分退款 | 退款金额不能超过可退余额，超出则拒绝 | ⚠️ 部分 | analysis §4 · `src/service/refund_service.py:110` |
| REQ-REFUND-04 | P0 | 调第三方退款 | 调支付中心执行真实退款 | 退款请求交由支付系统执行真实退款，失败会自动重试 | ✅ 已实现 | analysis §7 · `src/clients/payment_client.py:55` |
| REQ-REFUND-05 | P0 | 更新订单状态 | 退款成功后置订单为已退款 | 退款成功后，订单变为"已退款"，资金原路退回 | ✅ 已实现 | analysis §6 · `src/repo/order_repo.py:120` |

#### 模块：可靠性
| REQ id | 优先级 | 功能 | 简述 | 验收标准（业务语言，可判定） | 实现状态 | 实现锚点 |
|--------|--------|------|------|------------------------------|----------|----------|
| REQ-REFUND-06 | P0 | 退款幂等 | 同一订单重复/并发退款不双退 | 同一订单的并发或重复退款，只有一次能成功 | ✅ 已实现 | analysis §8 · `src/service/refund_service.py:160` |
| REQ-REFUND-07 | P0 | 失败补偿 | 第三方退款失败后兜底 | 第三方退款失败时，系统记录失败并兜底重试，不直接向用户报错 | ✅ 已实现 | analysis §5 · `src/service/refund_service.py:138` |
| REQ-REFUND-08 | P1 | 部分退款拆单 | 一笔订单分多次部分退 | 是否支持一笔订单分多次部分退，规则未明确 | ❌ 缺口 | analysis §8 ⚠ 未确认 |

### 5. 明确不做（Out of scope）

- 不做商家代发起退款：退款入口仅 C 端用户，无商家后台入口 —— analysis §2。
- 不做退款审批流：满足可退条件即自动退款，无人工审核环节 —— analysis §4 主流程。

### 6. 非功能约束

- 时效：退款调用第三方超时阈值 3s —— analysis §7。
- 并发：同一订单并发退款须串行化、不双退 —— analysis §8。
- 可靠性：第三方失败重试 2 次，仍失败入补偿队列 —— analysis §7。

### 7. 实现与需求偏差

- **过度实现 / 副作用**：退款失败仅入补偿队列、不向用户报错 —— 用户体验上用户可能长时间不知退款失败，需求层面是否需要异步通知用户 ⚠ 未确认 —— analysis §5。
- **需求缺口**：补偿队列的重试上限与告警未在本业务内，依赖外部 job —— 一笔退款可能因补偿 job 配置不当而长期挂起 ⚠ 未确认 —— analysis §8。

### 8. 未决问题 / 已知缺口

- **部分退款拆单规则**：`refund_service.py:110` 金额计算未体现是否支持一笔订单分多次部分退，需找产品 / 历史 PR 确认 —— 见 analysis 已知缺口。
- **补偿逻辑归属**：失败后入补偿队列，但补偿 job 不在本业务代码内，未追踪其重试上限 —— 见 analysis 已知缺口。
