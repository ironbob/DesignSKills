# 技术方案 / 设计文档（rfc）

## 匹配信号
重大方案/技术选型需评审。读者动作：评审通过/否决/修改方案。强调"权衡"而非"结论"。

## 必备章节（slot）与每节必含

| slot | 标题示例 | 每节必含 |
|------|----------|----------|
| `context` | 背景与目标 | 解决什么问题、为什么现在做、目标（可度量） |
| `current_state` | 现状与约束 | 现有做法、痛点、技术与资源约束 |
| `proposal` | 提案 | 推荐方案的设计、关键决策（`[!decision]` 带理由） |
| `alternatives` | 备选方案与权衡 | **至少一个备选** + 各自利弊 + 为何不选（R-W3） |
| `decision` | 决策 | 最终决定 + 决策依据；未决项标"开放问题" |
| `impact` | 影响与风险 | 迁移成本、兼容性、风险与缓解 |

> `alternatives` 是 RFC 的灵魂：没有备选的"方案"只是声明，不是评审材料。

## 常见缺陷
- 只讲推荐方案，不讲为什么不是别的（缺备选/权衡）。
- 目标不可度量（"提升性能"而非"P99<200ms"）。
- 决策藏在正文里，评审者抓不住结论。
- 只谈收益不谈迁移成本与风险。

## 专业示例片段
````markdown
<!-- doc:section slot=proposal intent=推荐方案与关键决策 -->
## 提案
采用"自适应限流 + 连接池弹性扩缩 + 降级"组合方案。

> [!decision] 限流算法选令牌桶
> 理由：需支持突发流量且可平滑限速。备选：漏桶（不支持突发，否）；计数器（粗粒度，否）。

<!-- doc:section slot=alternatives intent=权衡为何不选备选 -->
## 备选方案与权衡
| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 组合方案（推荐） | 多层防御，根治 | 实现复杂 | ✅ 采用 |
| 仅扩容连接池 | 改动小 | 治标，峰值再翻倍仍打满 | ❌ |
| 降级非核心链路 | 快 | 影响收入功能 | ❌ 作兜底 |
````
