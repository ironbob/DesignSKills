# 复盘报告（postmortem）

## 匹配信号
事后学习、驱动改进落地。读者动作：相关人认领 Action 并执行。**非追责、非对外公关。**

## 必备章节（slot）与每节必含

| slot | 标题示例 | 每节必含 |
|------|----------|----------|
| `summary` | 摘要 | 一句话发生了什么 + 严重程度 + 当前状态（`[badge:L]`） |
| `impact` | 影响 | 受影响范围：用户数/订单数/时长/资损（引用 `datasets`） |
| `timeline` | 时间线 | 从首个告警到完全恢复的关键节点（用 `timeline` 块） |
| `root_cause` | 根因 | **系统性根因**（流程/架构/监控），不停在"直接原因"；因果链显式 |
| `actions` | 改进项 | 每条 Action：内容 + **Owner + DDL + 验收标准**（表格最佳） |
| `lessons` | 经验沉淀 | 可复用的应急动作、监控/流程补齐方向 |

> `actions` 的 Owner/DDL 缺失是最常见缺陷——必须可追责可验收。

## 常见缺陷
- 只罗列"发生了什么"，不说"为什么"和"所以呢"。
- Action 没有 Owner / DDL，开完会就忘。
- 变成追责大会，没人敢说真因。
- 根因停在直接原因（"连接池打满"），未追问系统性原因（"为什么没有限流"）。
- 时间线缺关键节点（首个告警/介入/恢复）。

## 专业示例片段
````markdown
<!-- doc:section slot=impact intent=量化事故影响 -->
## 影响
本次事故影响 {{d:affected_orders}} 单支付订单，持续 {{d:duration_min}} 分钟，估算资损 {{d:revenue_loss}}（口径见 datasets）。严重程度定级 P1。

<!-- doc:section slot=root_cause intent=论证系统性根因 -->
## 根因
直接原因：流量峰值并发达 {{d:peak_concurrency}}，连接池上限 50 被打满 → 请求排队 → 超时级联 → 用户支付失败。
系统性根因：① 缺少自动限流，峰值流量直达下游；② 连接池容量未随流量弹性扩缩；③ 监控只有"连接池使用率"，无"即将打满"的提前告警。

<!-- doc:section slot=actions intent=可追责可验收的改进项 -->
## 改进项
| Action | Owner | DDL | 验收 |
|--------|-------|-----|------|
| 接入自适应限流 | 张三 | 6/28 | 峰值并发>阈值时限流生效，超时率<0.1% |
| 连接池弹性扩缩 | 李四 | 7/05 | 压测并发 2x 不打满 |
| 增加提前告警 | 王五 | 6/20 | 使用率>80% 提前 5 分钟告警 |
````
