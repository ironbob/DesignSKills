# 产品需求文档（prd）

## 匹配信号
定义产品功能"做什么"。读者动作：确认开发范围、对齐目标。强调"问题与可度量成功"。

## 必备章节（slot）与每节必含

| slot | 标题示例 | 每节必含 |
|------|----------|----------|
| `problem` | 问题与用户价值 | 谁的什么问题、现状痛点、不做的影响 |
| `goals` | 目标 | 本期要达成什么（与问题对应） |
| `success_metrics` | 成功指标 | **可度量**指标 + 目标值 + 衡量方式（引用 `datasets`） |
| `scope` | 功能范围 | 功能清单（分组 + 优先级 P0/P1/P2 + 简述 + 验收） |
| `out_of_scope` | 明确不做 | 划清边界，防蔓延 |

> `success_metrics` 必须可度量："下单转化率从 {{d:conv_before}} 提升到 {{d:conv_target}}"，不是"体验更好"。

## 常见缺陷
- 成功指标不可度量（"用户更满意"）。
- 只列功能不说要解决什么问题。
- 没有 out_of_scope，范围无限蔓延。
- 功能无优先级、无验收标准。

## 专业示例片段
````markdown
<!-- doc:section slot=success_metrics intent=可度量的成功判据 -->
## 成功指标
- 下单转化率：{{d:conv_before}} → {{d:conv_target}}（埋点，上线 30 天）
- 下单耗时：{{d:order_time_before}} → {{d:order_time_target}}（端上埋点）

<!-- doc:section slot=scope intent=分组的功能与验收 -->
## 功能范围
#### 模块：下单流程
| 优先级 | 功能 | 验收 |
|--------|------|------|
| P0 | 步骤精简 5→3 | 关键路径 ≤3 步 |
| P1 | 地址自动填充 | 命中率≥80% |
````

> 注：PRD 的"功能范围"与 `clarify-requirements` 的功能清单同源；若上游已有需求文档，可直接引用为 `sources`，本节聚焦产品层。
