# 架构决策记录（adr）

## 匹配信号
记录一个架构/技术决策的"为什么"，供后人查阅。读者动作：理解并认可决策依据。**短而精，一个决策一篇。**

## 必备章节（slot）与每节必含

| slot | 标题示例 | 每节必含 |
|------|----------|----------|
| `context` | 背景 | 决策面对的情境、约束、要解决的问题 |
| `decision` | 决策 | 最终决定（一句话可复述） |
| `rationale` | 理由 | 为什么这么定；考虑过的因素；至少一个备选与为何不选 |
| `consequences` | 后果 | 正面/负面影响、需跟进的事项 |

> ADR 的价值在 `rationale`：后人要知道"当年为什么这么选"，否则决策成黑盒。

## 常见缺陷
- 只写"决定用 X"，不写为什么。
- 漏 `consequences`（决策的代价没人知道）。
- 一篇塞多个决策（拆成多篇）。
- 决策已过时却不标记（加 status：proposed/superseded）。

## 专业示例片段
````markdown
<!-- doc:section slot=decision intent=一句话可复述的决策 -->
## 决策
支付核心采用事件溯源（Event Sourcing）写模型。

<!-- doc:section slot=rationale intent=为什么这么定 -->
## 理由
读多写少、需完整审计链与时间旅行查询。备选：CRUD + 审计日志（审计不完整、难回放，否）。权衡：写路径复杂度上升，可接受（支付写量远小于读量）。

<!-- doc:section slot=consequences intent=决策的代价与跟进 -->
## 后果
- 正面：天然审计、可回放、读写分离清晰。
- 负面：写模型复杂度↑、需 saga 处理最终一致。
- 跟进：建立事件 schema 版本治理（Action：赵六，Q3）。
````
