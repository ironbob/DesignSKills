# 架构可读性评估（模块 C）

> 配合 `arch-quality-eval` 的 Checklist 第 5 步使用。评估「**结构是否让人一眼看懂职责与依赖**」——只评**架构层**可读性，**不包含 lint 能查的代码风格**（PRD §4-C / §5）。结论写进 `findings.json` 的 `readability` 四轴 + report「架构可读性」节。

## 一、可读性 ≠ 代码风格（最重要的划界）

| ❌ 不评（lint/代码风格范畴，明确排除） | ✅ 评（架构可读性范畴） |
|----------------------------------------|----------------------|
| 缩进、空行、括号风格 | 模块/包/类**职责是否一眼可懂** |
| 命名**格式**（驼峰/下划线/大小写规范） | 命名**是否表意**（语义层：名字是否反映职责，如 `OrderManager` 是否表意） |
| import 顺序、行宽 | **依赖关系是否可理解**（依赖谁、为何依赖、方向是否合理） |
| 注释格式、文件头模板 | **分层是否清晰**（层边界、依赖方向、是否穿透） |

判断准则：**换个格式化工具就能修的，不是本轴的事**；需要理解结构、职责、依赖才能判断的，才是。

## 二、四轴评估（对应 findings.json `readability` 四字段）

每轴：一句话结论（清晰/一般/混乱 + 具体表现）+ 至少一个证据锚点 file:line。结论要具体可核，不写「可读性好/差」这种空话。

### 1. responsibility_clarity 职责清晰度
- **问**：看模块/包/类的名字和结构，能不能一眼说出它干什么？有没有职责混杂（一个类干 N 件不相关的事）？
- **信号**：类名与实际职责不符；一个包里混着多个不相关子领域；核心类职责描述要写很长。
- **证据**：类/包定义 file:line + 职责混杂的具体表现。

### 2. dependency_understandability 依赖可理解性
- **问**：依赖关系能不能看懂——依赖谁、为什么依赖、方向合不合理？有没有「为什么它会依赖这个？」的费解边？
- **信号**：出人意料的依赖（底层依赖上层、无关模块互依赖）；依赖图杂乱无方向；隐藏依赖（反射/字符串配置）。
- **证据**：依赖边 file:line（import/引用）+ 为何费解的说明。

### 3. naming_expressiveness 命名表意度（语义层）
- **问**：包/类/模块名是否**反映其职责**？（不是格式规范，是表意）
- **信号**：`Manager`/`Helper`/`Util`/`XXXServiceImpl` 这类不表意的万能名；名字与职责矛盾（`OrderQuery` 里写了写操作）。
- **证据**：命名 file:line + 为何不表意。
- **注意**：这是**语义层**——「这名字让人误会职责」才报；「这名字不符合驼峰规范」不报（那是 lint）。

### 4. layering_clarity 分层清晰度
- **问**：分层清不清楚？层边界、依赖方向是否一眼可辨？有没有穿透/反向让人困惑？
- **信号**：层边界模糊（看不出哪层）；跨层穿透；层职责串味（Service 里全是 Web 逻辑）。
- **证据**：分层/穿透 file:line。

## 三、可读性问题 → finding

可读性问题**可以**作为 finding 报（`axis: readability`），尤其当它达到 major 影响（严重职责混杂/依赖不可理解，抬高上手与改动成本）：

- `category`：取 `responsibility-mixed` / `dependency-opaque` / `naming-unexpressive` / `layering-unclear` 之一。
- `principle_violated`：可读性轴 finding 仍要点一条原理（如「单一职责」「信息隐藏」「单向依赖」），或当它主要影响可理解性时点「可理解性/认知负载」（但须结合一条实在原理）。
- 证据、定级、优先级同其他 finding。
- **轻微的**可读性问题（个别命名不表意、个别费解依赖）可不单列 finding，只写进四轴结论 + 标 minor；**严重的**单列 finding 以便进优先级。

## 四、写进 report「架构可读性」节（模板对齐 `report-template.md`）

- 四轴逐条结论 + 证据。
- 一个总体可读性判断（清晰/一般/混乱）+ 一句最影响可读性的点。
- C++ 上若结构信息受限导致某轴判断保守，注明。

## 五、自检

1. **没混入 lint** —— 全节没有「缩进/格式/驼峰规范/import 顺序」类评判；只有语义/结构/职责/依赖。
2. **结论可核** —— 每轴有 file:line 锚点，不空泛。
3. **严重才单列** —— 轻微可读性问题不刷 finding 凑数，只进四轴结论。
