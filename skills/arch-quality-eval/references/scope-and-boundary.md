# 范围界定与边界（模块 A）

> 配合 `arch-quality-eval` 的 Checklist 第 1、2 步使用。这一步圈定「**评估哪一块代码**」——边界定错，后面全错。**范围未经用户确认前不深挖、不产出**（HARD-GATE）。

## 一、模块 A 三件事（PRD §4-A）

1. **模块范围界定**（P0）：接受用户指定的 feature/子系统根路径（可跨多目录），圈定评估边界。
2. **语言与结构识别**（P0）：识别 JVM(Java/Kotlin) / C++，及其包/目录/命名空间结构；C++ 明确标注能力受限。
3. **项目规约喂入**（P1）：**可选**接受用户**手工输入**的项目架构规约；未喂入则跳过规约检查，只跑通用准则。

## 二、怎么圈范围

### 2.1 输入形态
- 用户给一个或多个**根路径/目录/包**（如 `src/main/java/com/x/order/` + `src/main/java/com/x/promotion/`）。
- 可能给一个 **feature 名**（如「订单退款」）+ 路径；路径是主，feature 名是标签。

### 2.2 圈出覆盖文件集合（`covered_files`）
- 用 `rg --files` / `find` 枚举路径下源文件（`.java`/`.kt`/`.h`/`.hpp`/`.cc`/`.cpp`），剔除测试/resource/build 产物（除非用户明确含）。
- **跨多目录**：把用户给的多个根合并成一个覆盖集合。
- 结果写进 `findings.json` 的 `covered_files`（**非空**，显式声明优于推断——防「声称评估了其实没看」）。
- 文件过多时：用 `analysis-protocol.md §六` 的聚焦策略，但在 `covered_files` 里**声明覆盖集合**（即便精读的是热点子集，覆盖边界仍是全集，抽样部分在「已知缺口」说明）。

### 2.3 语言识别
- 按文件后缀 + 内容判定：`.java`/`.kt` → **JVM**；`.h`/`.hpp`/`.cc`/`.cpp`/`.cxx` → **C++**。
- 混合语言模块（少见）：以**主体语言**为 `language`，另一语言在「已知缺口」注明覆盖有限。
- C++ → 设 `cpp_limitation_noted: true`，并在 report 标注结构/依赖分析能力受限。

### 2.4 结构识别（粗读，不深挖）
- JVM：读 `package` 声明 + 目录，识别包/层结构（controller/service/dao…）。
- C++：识别命名空间 + 目录 + `#include` 大致图（粗）。
- 这一步只够划边界，**坏味道深挖在第 4 步**（确认范围后）。

## 三、项目规约喂入（可选，PRD §4-A P1）

- **主动问一句**：「有没有要对照的项目架构规约？如分层规则 / 模块边界 / 禁止依赖方向？有就贴给我（文本即可，不用文件）。」
- 喂入 → 解析成 `convention_rules: [{id, rule}]`（格式见 `convention-checking.md`），设 `conventions_fed: true`，第 4 步走规约检测。
- 不喂入 → `conventions_fed: false`、`convention_rules: []`，**规约检测整节不触发**（`validate_report.py` R-V 卡：不得出现规约违规结论）。
- **不自动解析** `ARCHITECTURE.md` 等文档（PRD §5）；**不从代码推断「应有架构」**（PRD §5）——规约只来自用户手工喂入。

## 四、两段式确认（HARD-GATE，第 2 步）

把下面四样呈现给用户确认，**未确认不进第 3 步**：

1. **路径 + 覆盖文件集合**（`covered_files` 概要：N 个文件 / 涉及哪些包或目录）。
2. **语言**（JVM / C++）+ 结构识别结果；C++ 标注能力受限。
3. **是否喂入规约**（喂入则附 `convention_rules` 清单；未喂入则明说「只跑通用准则」）。
4. **一句话模块职责**（这个模块是干什么的，作为评估基线；不是「应有架构」，只是「它声称做什么」）。

> 最大风险是「圈错模块 / 把无关代码拉进来或漏掉关键包」。这一步专门拦它。用户修订后回到本步重确认。

## 五、边界越界识别（scope 判定）

- **用户要「全 repo 巡检 / 全景图」** → 超出本 skill（PRD §5 巡检场景，非单模块诊断）；建议收窄到某 feature/子系统，或说明本 skill 不做巡检。
- **用户给的范围过大**（整个大系统）→ 提示「单模块诊断在大范围上 token/噪音压力大，建议聚焦到子系统/feature 级」，用聚焦策略（§2.2 / `analysis-protocol.md §六`）。
- **范围过小到单文件** → 可评估，但提示「单文件级架构味道有限，建议至少到包/子系统级才有架构意义」。

## 六、划界：不做什么（PRD §5）

- ❌ 不做全 repo 扫描 / 架构巡检全景图。
- ❌ 不自动解析项目架构文档（ARCHITECTURE.md 等）——规约只手工喂入。
- ❌ 不从代码结构推断「应有架构」当标准。
- ❌ 不读项目已有 ArchUnit / dependency-cruiser 等架构测试断言（PRD §5）。
