# 代码读取取证协议（模块 B/C 的证据来源）

> 配合 `arch-quality-eval` 的 Checklist 第 3 步使用。这一步解决「**怎么把架构结构与依赖事实读出来当证据**」——坏味道/可读性结论的 `evidence` 全来自这里。**对标 `code-analyze-business` 的 `coverage-strategy.md`**：LSP/rg 双轨，先检测→用足→不可用对话建议安装等回复→rg 降级。
>
> 本 skill 不内置完整 code-parsing 工具链；代码读取由 agent 用 LSP/rg 完成，证据回链 file:line。证据位置最后由 `validate_evidence.py` 做轻量真实性校验（文件存在、行号范围、note 关键字命中）。

## 一、要读出的几类事实

坏味道/可读性判定需要的证据，归为四类事实，逐类取证：

| 事实类 | 喂给哪些坏味道/可读性轴 | 怎么读 |
|--------|----------------------|--------|
| **包/模块/层结构** | 循环依赖、跨层、分层清晰度、God Package | 包声明/目录约定/命名空间 |
| **类型引用/依赖边** | 循环依赖、跨层、不稳定依赖、特性依恋、依赖可理解性 | import / 全限定类型引用 / `#include` |
| **类/包规模与职责** | God Class/God Package、职责清晰度 | 类定义 + 方法/字段清单 + 行数 |
| **可见性/暴露面** | 不恰当暴露、门面滥用 | 可见性修饰符 + 方法体 |

## 二、LSP / rg 双轨取证协议（严格按序）

符号/引用事实优先 LSP（精确），没有则降级 rg 文本搜索。

1. **先检测**：对一个代表性源文件做廉价 LSP 探测（如 `documentSymbol` on 一个核心类文件）。
   - 返回结果 → **LSP 可用**，走 §三。
   - 报错（无 language server / 未就绪）→ **不可用**，走第 3 步。
2. **可用时用足**（§三）：对核心类、接口/抽象、关键依赖边用 `incomingCalls`/`outgoingCalls`/`findReferences`/`goToImplementation` 取证。
3. **不可用时默认降级**：不主动安装，不停下来等用户。用第 4 步 `rg` 降级继续分析，并在 report「评估方法与已知缺口」标注「LSP unavailable，引用覆盖为文本搜索降级」。
4. **用户要求高精度时才等待**：如果用户明确要求必须用 LSP 精确查引用，再在对话里给一句安装建议（如「装 jdtls/clangd 即可用 LSP 精确查引用」）并等待用户处理；否则不阻断诊断。
5. **降级取证**：无 LSP 时用 `rg` 搜全限定类型名 / `import` / `#include` / 命名空间；疑似关系但无法实锤的 finding 标 `unconfirmed: true` 并登记缺口。

> 原则：能用精确符号查询就用；不能用时自动降级并留痕。装不装语言服务器是用户的环境决策，不让默认诊断卡在工具安装上。

## 三、LSP 可用时的取证清单

| # | 目标 | LSP 操作 | 拿到什么 |
|---|------|----------|----------|
| 1 | 核心类/接口 | `incomingCalls` + `outgoingCalls` | 入度/出度（God Class 判定 + 依赖中枢） |
| 2 | 跨包类型引用 | `findReferences` | 谁依赖它（循环依赖/跨层/特性依恋的边） |
| 3 | 接口/抽象 | `goToImplementation` | 几个实现、主链路走哪个（门面/分层判定） |
| 4 | 关键字段/可见性 | `documentSymbol` | 字段/方法可见性（不恰当暴露） |

无 LSP 时逐行换成「rg 搜符号名 → 人工判调用/引用关系」，节首标注降级。

## 四、JVM 取证要点（Java/Kotlin）

- **包结构**：`package` 声明 + 目录约定（`com.x.order.*`）。
- **依赖边**：`import` 语句 + 全限定类型引用（字段类型/参数类型/继承/方法调用）。Kotlin 同理（`import` + 全限定名）。
- **层识别**：按包/目录约定（controller/service/dao/domain/infra…）；**层定义以用户规约为准**（若喂入），否则按社区惯例命名 + 保守判断。
- **可见性**：`public`/`protected`/`private`/`package-private`（Kotlin: 默认 public，`internal`/`private`）。
- **规模线索**：类行数/方法数（rg/读文件），**只当线索**，不设绝对阈值判定 God Class。

## 五、C++ 取证要点（能力受限，必标）

C++ 无 package、依赖靠 `#include` / 命名空间 / `compile_commands.json`，**分析能力弱于 JVM**（PRD §6）：

- **结构**：用**命名空间 + 目录**近似「包」；`#include` 图近似依赖边。文本级 `#include` 易受前向声明/宏/PIMPL 干扰，结论**更保守**，疑似标 `unconfirmed`。
- **循环依赖**：`#include` 相互包含或命名空间循环——文本级能发现直接互 include，但间接环/宏引入的环可能漏，标注降级。
- **规模/可见性**：类规模（行数/方法数）、`public:`/`private:` 区段仍可判（rg）。
- **分层**：C++ 层边界常靠目录约定、不强制，跨层判定偏保守并标注。
- **`compile_commands.json`**：若项目根有该文件，可用其解析翻译单元的包含路径（仍属文本级，能力有限）；没有则纯 `#include` + 命名空间。
- **必登记**：report「已知缺口」须有一条「C++ 结构/依赖分析受限，部分坏味道（尤其循环依赖/跨层）覆盖深度低于 JVM，结论偏保守」。

## 六、大模块聚焦策略（PRD §6 噪音/聚焦）

feature/子系统级范围偏大，全量精读会 token 爆炸 + 噪音淹没关键问题。**热点优先**：

1. **先找热点**：按规模/引用密度定位候选热点（大文件、被很多类依赖的类、import 很多类型的类）——这些最可能是 God Class / 中枢 / 循环依赖参与者。
2. **聚焦热点取证**：热点类/包深度取证（§三/§四/§五）；非热点抽样确认「无重大问题」即可，不逐行精读。
3. **分层扫描**：先读包/层结构（粗），锁定可疑层/包，再下沉到类/方法（细）。
4. **不淹没**：report 只报**真实架构问题**，不罗列每个文件的细枝末节；噪音问题不进 finding。

> 聚焦策略的产出是「在有限 token 内抓到关键架构问题」；在 report「评估方法/已知缺口」节简述本次用了哪种聚焦 + 舍弃了什么（防「看似全覆盖实则抽样」的误导）。

## 七、防臆造（底线）

- 每条证据回链真实 `file:line`；rg/LSP 命中**只当线索**，是否真构成坏味道要看代码确认（注释/命名可能过时，以代码为准）。
- 推断但没实锤的标 `unconfirmed: true` 并登记「已知缺口」，不混进正文当事实。
- **禁止编造证据**：找不到定位点的结论必须标「未能定位」，不得编 file:line。
