# 架构坏味道目录（模块 B 主轴）

> 配合 `arch-quality-eval` 的 Checklist 第 4 步使用。本文件是「架构坏味道」检测的**条目集与客观判定标准**（PRD §4-B / §7 开放问题）。每条含：**定义 → 客观判定信号（可回链取证）→ 违反的通用设计原理 → 典型严重级**。
>
> **边界铁律**：只收**架构层**坏味道（结构/依赖/职责/分层），**不收 lint 能查的代码风格**（命名格式、缩进、空行、import 顺序）。命名只在「是否表意（语义层）」时进 `readability` 轴，不进本目录。

## 一、怎么用这个目录

1. **逐条对照、二分显式**：report「坏味道清单」节对每条核心坏味道给出 **已检出 / 未检出（N/A）** 二分判定（`validate_report.py` R-C 硬卡核心类齐全）。已检出的进 finding；未检出的也要显式标「未检出」，证明逐条过了一遍（防漏检）。
2. **判定必须有证据**：每条「已检出」都挂 `evidence`（file:line / 类 / 函数 / 依赖边）。**没有证据不得判「已检出」**；疑似但没实锤的标 `unconfirmed: true`。
3. **点名原理**：每条 smell finding 必须点名违反的**通用设计原理**（见每条「违反原理」字段），不空泛说「设计不好」。
4. **不逐条对照 SOLID**（PRD §5 out-of-scope）：原理通过「这条坏味道违反了哪条原理」间接体现，不单独跑 SRP/OCP/LSP/ISP/DIP 合规检查。

## 二、通用设计原理速查（finding 的 `principle_violated` 取这些）

- **高内聚低耦合** —— 职责内聚、依赖最小化
- **单向依赖原则** —— 依赖应单向无环
- **依赖倒置（DIP）** —— 高层不直接依赖低层细节，依赖抽象
- **稳定依赖原则（SDP）** —— 依赖应指向更稳定（更难变）的方向
- **分层不穿透** —— 上层依赖下层，不跨层、不反向
- **接口隔离** —— 不强迫依赖用不到的东西
- **信息隐藏** —— 只暴露必要的，内部实现封闭
- **单一职责** —— 一个模块/类只有一个变化的理由

## 三、核心坏味道（P0，必检）

### 1. 循环依赖 circular-dependency
- **定义**：包/模块/类之间形成依赖环（A→B→A）。
- **判定信号**：
  - JVM：包 A 的类 import 包 B 的类型，同时包 B 的类 import 包 A 的类型（取证 `import` / 全限定类型引用；LSP `incomingCalls`/`findReferences` 或 rg 搜全限定名）。
  - C++：头文件相互 `#include` 或命名空间间循环引用（能力受限，文本级 `#include` 图 + 命名空间推断，标注降级）。
- **违反原理**：单向依赖原则（亦违反稳定依赖）。
- **典型严重级**：**critical**（包级循环阻塞独立编译/部署/演进，是重构主要阻塞点）。
- **注意**：方法/类级临时环未必 critical；**包/模块级环**才是 critical。

### 2. God Class / God Package god-class | god-package
- **定义**：单个类（God Class）或包（God Package）承担过多职责、成为系统中枢，大量其他类依赖它或被它依赖。
- **判定信号**：
  - 类行数/方法数/字段数显著高于同模块同类（取相对值，不设绝对阈值；取证类定义 file:line + 方法清单）。
  - 入度/出度过高（被很多类依赖 / 依赖很多类）—— LSP `incomingCalls`/`outgoingCalls` 或 rg 搜引用计数。
  - God Package：一个包内类过多、且包内职责混杂（多组无关注联的类共存）。
- **违反原理**：单一职责、高内聚低耦合。
- **典型严重级**：核心类 → **critical**；非核心类 → **major**。
- **注意**：不靠纯数值（LCOM/扇出）判定（PRD §5 out-of-scope 不做模块化度量）；数值只当**线索**，最终看职责是否真的混杂 + 是否是模块中枢。

### 3. 跨层调用 cross-layer
- **定义**：违反分层，出现跨层或反向依赖（如 Controller 直接调 Dao、Dao 调 Controller、领域层直接依赖 Web 层）。
- **判定信号**：
  - JVM：按包/命名约定识别层（controller/service/dao/domain/web…），找跨层 import（Controller import dao、domain import web）。
  - 取证 `import` 语句 file:line + 双方所属层。
- **违反原理**：分层不穿透（亦违反单向依赖）。
- **典型严重级**：贯穿核心链路的硬跨层 → **critical**；可插防腐层解的 → **major**。

### 4. 霰弹式修改 shotgun-surgery（变体见 §四 divergent-change）
- **定义**：一类改动要散落到很多地方才能完成（一个变更动 N 个类/包）——架构上职责被切开。
- **判定信号**：
  - 历史线索：同一改动主题（如「加一种促销」「改退款规则」）需要改 ≥3 个无继承关系的类/包（取证多个 file:line）。
  - 多个类因同一业务概念被迫协同改动。
- **违反原理**：单一职责、高内聚。
- **典型严重级**：**major**（显著抬高变更成本，但不一定阻塞）。

### 5. 不恰当暴露 inappropriate-exposure
- **定义**：本应封闭的实现被过度暴露（public 过宽、内部细节当 API、包把内部类全公开）。
- **判定信号**：
  - JVM：大量 `public` 方法/类本应 `private`/`package-private`；内部实现类出现在对外 API 包。
  - 取证可见性修饰符 file:line。
- **违反原理**：信息隐藏、接口隔离。
- **典型严重级**：**minor**（个别）→ **major**（系统性，暴露面成为隐式 API 导致改动牵连）。

## 四、扩充坏味道（P1，检到即报，同样挂证据 + 点原理）

### 6. 门面控制器滥用 facade-controller-abuse
- **定义**：Controller 承担业务逻辑/编排过多，退化成「胖门面」，Service 层形同虚设或被绕过。
- **判定信号**：Controller 方法体含大量业务分支/直接调多个 Dao/跨领域编排；Service 被旁路。取证 Controller 方法体 file:line。
- **违反原理**：分层不穿透、单一职责。
- **典型严重级**：**major**。

### 7. 平行继承体系 parallel-inheritance
- **定义**：多组继承体系同步增减（加一个子类要同时加好几个体系的对应子类）。
- **判定信号**：多组继承树在结构上一一对应、同步演化；新增一个概念要动多处继承。取证多组子类 file:line。
- **违反原理**：高内聚低耦合、单一职责。
- **典型严重级**：**minor**（局部）→ **major**（体系级）。

### 8. 中间人过载 middle-man
- **定义**：类的大部分方法只是转发给别的类（delegate 过载），自身无职责。
- **判定信号**：方法体大多是「return x.foo()」式纯转发，占比高。取证转发方法 file:line。
- **违反原理**：单一职责、信息隐藏（过度抽象）。
- **典型严重级**：**minor**。

### 9. 发散式变化 divergent-change（霰弹手术的变体）
- **定义**：一个类因多种不同的变更原因被反复改动（一个类对多条变更轴敏感）——与霰枪式修改（一变多）互补：这里是一对多方向。
- **判定信号**：一个类被多种无关业务主题改动（git 历史线索 / 类内方法按多个无关主题聚类）。
- **违反原理**：单一职责。
- **典型严重级**：**major**。

### 10. 特性依恋（架构层）feature-envy-arch
- **定义**：架构层而非方法层——某模块/包的方法大量操作另一个模块/包的数据，像「更属于那边」。
- **判定信号**：A 包的类大量调用 B 包的 getter/数据访问、自身数据用得少。取证跨包数据访问 file:line。
- **违反原理**：高内聚低耦合、信息隐藏。
- **典型严重级**：**major**。

### 11. 不稳定的依赖方向 dependency-toward-instability
- **定义**：依赖指向了更容易变化的方向（稳定模块依赖了易变模块），导致稳定模块被连带扰动。
- **判定信号**：抽象/通用层依赖了具体/频繁变更层（取证依赖边 + 变更频率线索，后者往往需 git 历史，标 `unconfirmed` 若无实锤）。
- **违反原理**：稳定依赖原则（SDP）、依赖倒置（DIP）。
- **典型严重级**：**major**（依赖方向判断常带推断，谨慎标 unconfirmed）。

## 五、与「不做」的划界（防越界）

- ❌ **代码风格/lint**：命名格式（驼峰/下划线）、缩进、import 顺序、空行——**不收**。可读性只取「命名是否表意」语义层（见 `readability-assessment.md`）。
- ❌ **纯模块化度量**：LCOM/扇出/不稳定性/距主序列等纯数值——**不靠它们判定**，只当线索（PRD §5）。
- ❌ **独立 SOLID 逐条合规**：不单独跑 SRP/OCP/LSP/ISP/DIP 检查表；原理通过 finding 的 `principle_violated` 间接体现。
- ❌ **推断「应有架构」**：不把现状当标准反推「应该怎样」（PRD §5），只判「现状是否违反通用原理 / 用户喂入规约」。

## 六、C++ 特别说明

C++ 无 package 概念，依赖靠 `#include` / 命名空间 / `compile_commands.json`，**结构/依赖分析能力弱于 JVM**（PRD §6）。在 C++ 模块上：

- 「包级」概念用**命名空间 + 目录**近似；循环依赖按 `#include` 图 + 命名空间推断，能力受限，结论标 `unconfirmed` 更保守。
- God Class 仍可判（类规模 + 职责混杂 + 引用计数，靠 rg）。
- 跨层/分层判定弱（C++ 层边界常靠目录约定，不强制），结论偏保守并标注。
- 在 report「已知缺口」登记「C++ 结构/依赖分析受限，部分坏味道覆盖深度低于 JVM」。
