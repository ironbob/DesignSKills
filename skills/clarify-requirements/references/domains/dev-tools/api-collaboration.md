> 树路径：dev-tools > api-collaboration

> 使用前先读父类：[`README.md`](README.md)

## 匹配信号
- 关键词：接口管理 / API 文档 / Mock / 联调 / OpenAPI / Swagger / 接口测试 / API 协作。
- 场景：前后端接口约定、Mock 联调、接口文档沉淀。

## 继承要点（速览）
开发者工具通用全适用；本类的增量是——**Design-first + 可消费产物（Mock / SDK / 测试）**。

## 核心价值要素
让团队的接口"**定义即文档、文档即 Mock、变更可协同**"。不可缺：① 接口的集中定义与版本化 ② 一键生成 Mock 服务，前端不等后端就能联调。

## 业界标配功能
- 接口建模：请求 / 响应结构、字段类型、示例、校验规则。
- **一键 Mock**：按定义自动生成，支持规则（随机 / 枚举 / 延迟 / 状态码 / 条件分支）。
- 多环境与版本：dev / staging / prod，版本并存，变更历史与对比。
- 协同：团队空间、权限、评论、@提醒、变更通知。
- **多端消费**：导出 OpenAPI / Swagger，生成各语言 SDK / 请求代码 / Postman 集合。
- 调试与测试：在线请求、断言、用例集、轻量压测。
- **自动化入口**：CLI、API、CI 钩子（接口变更自动通知 / 挡板回归）。

## 业界标杆做法
- **Apifox / YApi / Stoplight / Postman / MSW**。
- **Design-first**：先定 OpenAPI 再写代码，文档与实现不脱节。
- **Mock 进前端构建链路**：MSW 在浏览器 / 测试里拦截，前端零等待。
- **契约测试（Pact）**：消费端与提供端用契约解耦，消除集成期返工。

## 常见陷阱
- 只做"文档"不做"可消费产物"（SDK / Mock / 测试）→ 文档过期、无人维护（接口文档类头号死因）。
- Mock 太"傻"（只回固定数据）→ 边界情况堆到集成期。
- 无版本 / 无变更通知 → 后端改字段，前端静默崩溃。
- 重 GUI 轻 API / CLI → 进不了 CI/CD，被绕开。
- 字段无校验规则 → 文档沦为"建议"而非"契约"。

## 本节点专属澄清
- 主要靠 GUI 还是要能进 CI / 脚本（API / CLI）？
- 要生成哪些**下游产物**（SDK / Mock / 测试用例 / OpenAPI）？
- 是否走 **Design-first**（先定义后实现）？
- 接口**版本与变更通知**怎么管？
- 要不要**契约测试**？
