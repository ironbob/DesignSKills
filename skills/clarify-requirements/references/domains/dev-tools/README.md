# 开发者工具 / 平台 领域知识

> 树路径：dev-tools
> 继承：无（根节点）
> 子类：`api-collaboration`（API / Mock 平台）等。命中下钻；否则回退到本节点。

## 匹配信号
- 用户是**开发者**；关键词：API / CLI / SDK / 插件 / 平台 / 流水线 / 调试 / 文档生成。
- 判据：工具是否需要被脚本 / CI / 其他工具调用。

## 核心价值要素
让开发者**更快、更可靠、更少摩擦**地完成某类开发任务。开发者对"摩擦"零容忍，差一点就不用。

## 业界标配功能（开发者强需求）
- **API / CLI 优先**：任何 GUI 能做的，有等价 API / CLI；能进 CI/CD、脚本、被其他工具调用。不能就被绕开。
- **可扩展性**：插件 / 钩子 / Webhook / SDK——能否被二次开发、嵌入他人工作流。"平台"与"一次性工具"的分水岭。
- **可观测性与可调试性**：日志 / 指标 / dry-run / 明确错误信息。开发者出问题第一反应是"我能不能自己查"。
- **文档与示例是一等公民**：quickstart、可运行示例、changelog。不是附属品，是产品本体。
- **幂等 / 可回滚 / 安全默认**：破坏性操作要 dry-run / 二次确认 / undo。
- **性能与资源占用**：开发者会测量；慢或重就被弃用。

## 分析侧重（开发者镜头）
- API / CLI 体验、可扩展性、可观测性、文档示例、幂等安全、性能。
- 标配来源：同类开源 / 商业工具、开发者社区共识、相关规范（OpenAPI / OCI / JSON Schema）。

## 业界标杆做法
- 看 GitHub 高星同类工具的 DX（开发者体验）。
- "约定优于配置"、合理默认值、零配置上手。

## 常见陷阱
- 重 GUI 轻 API / CLI → 进不了流水线，被绕开。
- 无可扩展点 → 用一次就到顶，沦为一次性工具。
- 错误信息含糊 → 开发者无法自助排查。
- 文档滞后 / 无可运行示例 → 上手成本高，弃用。
- 破坏性操作无 dry-run / 无回滚 → 不可信。

## 本节点专属澄清
- 主要靠 **GUI** 还是要能进 **CI / 脚本**（提供 API / CLI 吗）？
- 要不要**可扩展**（插件 / 钩子 / Webhook / SDK）？
- 错误与日志够不够开发者自助排查？
- 文档 / quickstart / 示例齐不齐？
- 有没有破坏性操作？是否 dry-run / 可回滚？
