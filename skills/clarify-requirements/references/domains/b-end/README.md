# B 端 / 企业内部工具 领域知识

> 树路径：b-end
> 继承：无（根节点）
> 子类：`itsm`（工单 / 服务台）等。命中下钻；否则回退到本节点。

## 匹配信号
- 给**企业 / 组织内部**用，多角色、有流程、要合规。
- 关键词：审批 / 工单 / 流程 / 权限 / 内部系统 / 后台 / ERP / CRM / OA。

## 核心价值要素
让组织内的某类**业务流转**有迹可循、可追踪、可度量、可管控。

## 业界标配功能（B 端强需求，C 端常漏）
- **角色与权限矩阵**：多角色（提交 / 处理 / 审批 / 管理），数据与操作按角色隔离。B 端头号盲区。
- **业务流程与状态机**：状态几种、谁能推动、卡住怎么办、能否回退 / 驳回。把状态图先画出来。
- **与现有系统集成**：数据来源与去向、通知（钉钉 / 企微 / 邮件 / SSO）、对接 CMDB / ERP。集成往往是最大工作量。
- **合规 / 审计 / 数据治理**：操作留痕（审计日志）、数据权限（行 / 列级）、合规要求（等保 / GDPR / 内控）。C 端常可后补，B 端常是硬门槛。
- **运营可配置性**：流程 / 模板 / 规则后台可配，避免每次改动找开发（"可配置系统"与"一次性脚本"的分水岭）。
- **报表与度量**：处理量、时长、超时率、积压。

## 分析侧重（B 端镜头）
- 角色矩阵、状态机、集成清单、合规审计、非功能（并发 / 数据量 / 可用性 / 安全 / 响应时长）。
- 标配来源更多是**行业规范与框架**（ITSM / ITIL、CRM / ERP 实践、合规框架），而非单个竞品。

## 业界标杆做法
- 服务台：ServiceNow、Jira Service Management（ITSM 框架）。
- 流程引擎 / 表单引擎可配置，降低二次开发。

## 常见陷阱
- 字段过多 → 义务劳动，放弃 / 填错。
- **无 SLA / 无升级** → 石沉大海，信任崩塌（B 端头号死因）。
- 状态机混乱（死状态 / 中间态过多）→ 流转卡住。
- 通知风暴 → 用户屏蔽。
- 只管"录"不管"度量" → 无法证明价值。

## 本节点专属澄清
- 有哪几类**角色**，各自能做什么？
- **状态流转**图长什么样？能否回退 / 驳回？
- 要对接哪些**系统**（SSO / 企微钉钉 / CMDB / ERP）？
- **SLA / 升级**要求？
- 有无**审计 / 合规**硬要求？
- 非功能硬指标（并发 / 数据量 / 响应时长）？
