# 工单 / IT 服务台 领域知识

> 树路径：b-end > itsm
> 继承：`b-end`（角色矩阵 / 状态机 / 集成 / 合规 / SLA 均适用）
> 使用前先读父类：[`README.md`](README.md)

## 匹配信号
- 关键词：工单 / 提单 / 服务台 / ITSM / 故障申报 / 客服工单 / 维修派单。
- 场景：员工提 IT 故障、客户报修、内部审批流转。

## 继承要点（速览）
B 端通用全适用；工单的增量是——**SLA 驱动的闭环流转 + 自动化分流**。

## 核心价值要素
让组织内的"**请求—响应**"有迹可循、可追踪、可度量。不可缺：① 闭环流转（提交 → 分配 → 处理 → 待验收 → 关闭）② 可追踪的状态与历史。

## 业界标配功能
- 多角色与权限：提单人 / 一线 / 二线专家 / 管理员。
- **SLA 与超时升级**：响应 / 解决时长阈值 → 自动催办 / 升级 / 转派。
- 分类与模板：按类型预填字段，降低填写成本。
- 通知与提醒：状态变更、@提及、即将超时（钉钉 / 企微 / 邮件）。
- 知识库关联：推荐相似工单 / 常见问题，减少重复提交。
- 报表与度量：处理量、平均时长、超时率、积压趋势。
- 审计日志：谁在何时改了什么，合规留痕。

## 业界标杆做法
- **ServiceNow / Jira Service Management / Freshservice**：ITSM 框架，"服务目录 + SLA + 知识库"三位一体。
- 模板预填、相似工单推荐、富媒体附件、移动端提单。
- **自动化分流**：按关键词 / 部门路由到处理组。

## 常见陷阱
- 字段过多 → 提单人放弃 / 填错（义务劳动，比 C 端更严重）。
- **无 SLA / 无升级** → 工单石沉大海（B 端头号死因）。
- 状态机混乱（中间态 / 死状态过多）→ 流转卡住、无处可退。
- 通知风暴 → 用户屏蔽，反而失联。
- 只管"提"不管"度量" → 系统被边缘化。
