# 澄清问题与边界控制

> 配合 clarify-requirements skill 的 Checklist 第 3-4 步使用。
> 核心纪律：**一次只问一个问题；多选优先；只问"做什么"，不问"怎么做"。**
> 本 skill 独立，只负责需求澄清，不涉及设计 / 技术实现。

---

## 一、澄清问题分类

按以下 6 类组织提问。先问"目的"和"用户"，再逐步收敛到"范围边界"。不要跳着问。

### 1. 目的类（最先问）
- 这个东西最终解决什么问题？没有它会怎样？
- 如果只能留一个最核心的能力，是哪个？
- 做完之后，怎么判断它"成了"？

### 2. 用户类
- 主要给谁用？他们现在怎么解决这个问题？
- 还有没有第二类用户？谁**不是**目标用户？

### 3. 场景类
- 用户在什么情境下会用？（时间 / 地点 / 设备 / 前后动作）
- 第一次用和熟练之后，用法一样吗？
- 多久用一次？一次用多久？

### 4. 约束类
- 有没有平台 / 时间 / 预算 / 合规上的硬限制？
- 有没有必须兼容的现有系统或数据？
- 有没有不能碰的红线（绝对不做某事）？

### 5. 成功标准类
- 上线后看哪个指标判断有效？（要可量化的）
- 什么程度算"够用"，可以停手？
- 有没有不能接受的失败情形？

### 6. 范围边界类（最后问）
- 哪些是你想过但决定先不做的？
- 有没有看起来相关、但其实不该塞进来的？
- 本期 MVP 最少要有什么，你才会真的用它？

## 二、好问题 vs 坏问题对照

| ❌ 坏问题（含糊 / 开放式过载） | ✅ 好问题（具体 / 聚焦 / 易答） |
|---|---|
| 你要什么功能？ | 这个 App 主要给谁用、什么场景下打开它？ |
| 需求都清楚了吗？ | "做完后用哪个指标判断它成了"——你能定一个吗？ |
| 有什么特殊要求？ | 有没有平台限制：只要手机，还是也要网页？ |
| 功能要多少？ | 本期 MVP 最少要有什么，你才会真的用它？ |
| 你觉得还需要什么？ | 我列了 A/B/C 三个先不做，你看哪个其实要做？ |

## 三、越界拉回话术

当用户（或自己的思路）滑向交互 / UI / 技术层时，用以下话术拉回，并记入"未决问题"。**不在本阶段展开这些话题。** 本 skill 只负责需求澄清，不涉及设计 / 技术实现。

### 交互类（页面 / 按钮 / 跳转 / 布局）
> "『首页怎么布局 / 按钮放哪 / 几个 Tab』是交互层的事，超出需求澄清的范围。我们只定『要哪些功能』。先记一笔到未决问题，留给后续的交互设计环节。"

### 技术类（架构 / 技术栈 / 数据模型 / API）
> "『用什么框架 / 数据库怎么设计 / 前后端怎么分』是技术方案的事，超出范围。这阶段只定功能。先记一笔到未决问题，留给后续的技术方案环节。"

### 视觉类（配色 / 字体 / 组件 / 动效）
> "视觉风格是设计环节的事，现在定太早。我们先把功能边界画清楚。"

### 通用拉回句式
- "这个问题很好，但它属于 [交互 / 技术 / 视觉] 决策，不在需求澄清的范围。"
- "我先记到未决问题。本 skill 到产出功能需求文档即结束，不涉及设计 / 技术实现。"

## 四、范围方案切分模板

当功能较多时，提出 2-3 档范围方案，**先说推荐项及理由**。

### 切分维度（按优先级 P0/P1/P2 归档）
- **MVP（最小可用）**：只含 P0。判据："没有这个，用户根本不会用它"。
- **标准版（首批上线）**：P0 + P1。判据："核心闭环完整，且 P1 里某项直接决定关键体验指标"。
- **完整版（后续迭代）**：P0 + P1 + P2。判据："没有也行，有了更好"。

### 呈现格式
```
建议三档范围：
- 【推荐】标准版（P0 + P1）：含 xxx 功能。
  理由：核心闭环完整，且 P1 里 <某项> 直接决定 <体验指标>，缺了用户会流失。
- MVP（仅 P0）：含 xxx。适合想最快验证想法 / 资源极度有限。
- 完整版（+ P2）：额外 xxx。适合确定要长期投入。
```

### 切分时的判断要点
- 默认推荐"标准版"，除非用户明确要快速验证（推 MVP）或明确要全量（推完整版）。
- 主动建议砍 P2（YAGNI）——明确告诉用户"这几个我觉得可以先不做"。
- 不替用户决定 P0/P1 边界，但可以质疑："你把 X 放 P0，但在 <场景> 下它似乎不是必需，降到 P1 行吗？"
