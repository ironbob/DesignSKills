# 领域分析：用专业与业界做法审视需求

> 配合 clarify-requirements skill 的 Checklist 第 2 步。
> 目的：在澄清需求**之前**，先用该领域的专业知识与业界通行做法做一遍基本分析，识别用户原始想法里的**盲区**（漏掉的标配）和**风险**（踩中的陷阱），让后续澄清建立在专业判断之上，而非盲目接受用户的初始表述。

---

## 一、为什么先做领域分析

用户的一句模糊需求往往只包含"他想到的"，而漏掉了该领域**几乎所有产品都有的标配**，也可能踩中**常见的坑**。若直接问"你还想要什么"，用户多半答不上来——他不知道自己不知道。

领域分析的价值是：用专业视角**主动指出**这些标配与陷阱，把它们变成可讨论的具体选项，再让用户取舍。这才是"澄清"而非"记录"。

---

## 二、分析维度

对该需求所属的领域/品类，逐项过一遍。不强求每项都深，但每项都要有结论（"有 / 无 / 不适用"）。

### 1. 核心价值要素
这类东西**本质**要解决什么问题？不可缺的那 1-3 个能力是什么？
> 例：背单词 App 的核心价值是"记住"，所以"间隔重复 / 复习调度"是不可缺的；没有它就只是个生词本。

### 2. 业界标配功能
这类产品/系统，**绝大多数**都有哪些功能模块？用户大概率预期有，但原始需求里没提。
> 例：电商 App 的标配——商品浏览、搜索、购物车、下单、支付、订单状态、售后。漏掉任何一项都不完整。

### 3. 典型用户旅程
一个用户从"第一次接触"到"达成目标"会经过哪些环节？每个环节需要什么功能支撑？
> 例：在线教育——发现课程 → 了解详情 → 开始学习 → 持续学习（进度 / 激励）→ 完成评估 → 复盘。每个环节对应一组功能。

### 4. 业界标杆做法
该领域公认做得好的产品/系统，在关键环节上是怎么处理的？有什么被验证过的实践？
> 例：Spaced Repetition（Anki）、即时反馈闭环（Duolingo）、降低启动阻力（首页"继续学习"大卡片）。

### 5. 常见陷阱与反模式
这类项目**经常踩的坑**是什么？哪些做法已被业界证明效果差？
> 例：学习类 App 在学习页塞社交 / 广告打断心流；结果页没有"下一步"形成死胡同；一次性铺题海而非单题聚焦。

---

## 三、如何呈现给用户

领域分析不是闷头做完就完，必须把发现**结构化地呈现给用户**，作为澄清的起点。

**关键：节点知识是假设，不是答案。** 加载节点后，对其每条标配 / 标杆 / 陷阱，对照用户的具体产品逐条判"**适用 / 调整 / 不适用（+原因）**"——只呈现过判断的结论，明显不符的直说"此产品不需要，因为…"。**禁止整段照搬节点原文**；每条结论都要连回用户提到的具体细节。**呈现前自检**：有没有整段照搬节点原文？每条结论是否连回用户具体细节？没有就重写。这才是"分析"，否则只是"念清单"。

```
基于这类产品（<领域/品类>）的专业与业界做法，我先梳理几点：

【核心价值】这类东西本质要解决 <X>，不可缺的是 <A / B>。
【业界标配（已适配）】这类通常有 <C / D / E>——对你的产品逐条判：<C 适用 / D 调整成… / E 不适用，因为…>。
【标杆做法】在 <关键环节>，业界（如 <标杆产品>）通行做法是 <做法>。
【常见陷阱】这类项目常踩的坑：<陷阱>，要注意避免。

请看这些是否符合你的预期，哪些要做、哪些先不做？
```

呈现后，把用户的回应作为 Checklist 第 3-4 步（用户场景、澄清问题）的输入。

## 四、何时需要联网搜索

领域分析**以既有专业知识为主**，多数情况下不需要联网。以下情形可辅以 web 搜索（如 WebSearch / web_search）补充业界做法：

- 领域较新或冷门，对该品类的标配 / 标杆把握不足。
- 需要具体的业界数据（市场规模、典型指标）来支撑判断。
- 用户提到的做法明显偏离常识，需核实是否已有成熟替代。

搜索时聚焦"业界通行做法 / 标杆产品 / 已知陷阱"，不做开放式调研；**基本分析**即可，不展开成深度研究报告。

## 五、领域知识树：定位最匹配节点 + 逐层回退

领域知识已**预置成一棵树**（见 `domains/` 目录）。做领域分析时，**先定位到最具体的节点、加载其知识；若该节点无内置知识，则回退到最近的祖先节点**。既有专业深度，又保证任何领域都不会"无知识可用"。

### 1. 树拓扑（节点 → 知识文件）

```
c-end  C 端通用                      domains/c-end/README.md                 ✅内置
├─ education  教育                   domains/c-end/education/README.md       ✅内置
│  ├─ k12  K12 学科                  domains/c-end/education/k12.md           ✅内置
│  ├─ language  语言学习             domains/c-end/education/language.md      ✅内置
│  └─ career  职业技能               domains/c-end/education/career.md        ✅内置
├─ tools  工具类                     domains/c-end/tools.md                   ✅内置
└─ games  游戏类                     (暂未内置 → 回退 c-end)                  ⬜占位

b-end  B 端 / 企业内部工具           domains/b-end/README.md                  ✅内置
└─ itsm  工单 / 服务台               domains/b-end/itsm.md                    ✅内置

dev-tools  开发者工具 / 平台         domains/dev-tools/README.md              ✅内置
└─ api-collaboration  API / Mock 平台  domains/dev-tools/api-collaboration.md ✅内置
```

> ⬜占位节点无需文件——命中时直接回退到父类。
> **新增一个 ✅节点**：按 `domains/_AUTHORING.md` 规约生成、过 `scripts/lint-node.sh` 校验，再把本表对应行 `⬜占位 → ✅内置`（补文件路径）。无需改任何逻辑。

### 2. 定位 + 回退算法

1. **定大类**：C 端 / B 端 / 开发者工具（看各根节点的"匹配信号"）。
2. **逐层下钻**：在大类下，沿子类看是否有更具体的节点匹配（看子节点的"匹配信号"）。
3. **取最深的有内置知识的节点**：
   - 命中 ✅节点 → 加载该文件；并按文件顶部"继承"声明，**一并加载祖先文件**，合并使用。
   - 该层是 ⬜占位 / 无匹配 → **回退到最近的 ✅祖先节点**。
4. **兜底**：最差也落到大类根节点（c-end / b-end / dev-tools），永不"无知识可用"。

> 每个知识文件统一结构：**匹配信号 / 继承要点 / 核心价值 / 业界标配 / 标杆做法 / 常见陷阱 / 专属澄清**。
> 继承维护：更新父类时，同步各子类文件顶部的"继承要点（速览）"。

### 3. 回退演示

用户说"**做个背单词 App**"：
→ 大类 **c-end** → 子类 **education**（命中"词汇 / 记忆 / 复习"）→ 教育下看更具体子节点：背单词 ≈ 语言学习，**命中 language ✅** → 加载 `language.md` + 父 `education/README.md` + `c-end/README.md`，合并分析。

用户说"**做个消除类小游戏**"：
→ 大类 **c-end** → 子类 **games**（命中"游戏"）但 games ⬜占位 → **回退到 `c-end/README.md`**，用 C 端通用镜头分析。

## 六、流程示例：对"做一个记账 App"做领域分析（演示定位 + 回退）

**定位**：记账 → 大类 **c-end**（C 端通用）→ 无金融 / 理财子节点 → **回退到 `c-end/README.md`**，用 C 端通用镜头 + 记账品类专业判断分析。

【核心价值】记账 App 本质是"让用户清楚钱花在哪、并能坚持记"。不可缺：① 快速记账（低摩擦录入）② 收支分类与统计。

【业界标配】（用户大概率没提但需要）：
- 多账户（现金、银行卡、信用）
- 周期性统计（日 / 周 / 月报表）
- 预算与超支提醒
- 数据导出 / 备份
- 分类自定义

【标杆做法】
- 录入极简：随手记、MoneyWiz 都把"3 秒记一笔"当核心指标。
- 自动化：账单导入、周期账单自动生成（银行 / 信用卡 App 通行）。
- 可视化：饼图 / 趋势图是标配。

【常见陷阱】
- 录入流程太长 → 用户坚持不下来（记账类最大流失原因）。
- 分类过细 → 维护成本高，用户放弃。
- 没有快速进入 → 结账时场景用不上。

→ 把以上呈现给用户："这些标配（多账户 / 预算 / 统计 / 导出）你需要哪些？录入要做到多快？" 以此驱动澄清。

### 6.2 适配判断演示：对"做一个 K12 少儿编程启蒙 App"做领域分析

**定位**：少儿编程 → c-end → education → **k12**（命中"少儿 / 编程 / 启蒙"）。加载 k12 + education + c-end。

**关键——不是念 k12.md，而是逐条适配**（节点是假设，对照"少儿编程"这个具体产品判）：

| k12.md 的标配 | 判定 | 原因 / 调整 |
|---|---|---|
| 教材版本（人教/北师大）/ 同步校内 | ❌ 不适用 | 编程启蒙无统编教材、非校内学科 |
| 题型 + 拍照搜题 / 批改 | ❌ 不适用 | 编程是项目 / 代码，不是题 |
| 错题本 | 🔄 调整 → 作品集 / 项目复盘 | 代码不是题，但"做过可回看重做"对应 |
| 知识点掌握度 | ✅ 适用（调整为编程概念：循环 / 条件 / 变量） | — |
| 家长端（学情 / 时长管控） | ✅ 强适用 | 少儿 + 护眼 / 屏幕时间 |
| 考前冲刺 / 真题 | ❌ 不适用（考级除外） | 无标准化考试 |

**适配后呈现给用户**（节选）：
- 【核心价值】让少儿在"拖拽创造"中建立编程思维，不是做题。
- 【标配（已适配）】图形化拖拽编辑器、按概念分级（序列 → 循环 → 条件 → 变量）、作品集、家长端；**不要**教材版本 / 校内同步 / 搜题。
- 【标杆】Scratch（图形化 + 社区）、Code.org（关卡课）、编程猫 / 核桃编程（体系课 + 辅导）。
- 【陷阱】太像"做题" → 失去兴趣；只看不练 → 学不会；无作品输出 → 没成就感。
- 【专属澄清】目标年龄（6-9 图形化 / 10+ 文本）？带辅导还是纯自学？效果看"能做出什么项目"还是考级？

> 对比：若**写死套用** k12.md，会硬塞"教材版本 / 校内同步 / 错题本"——对少儿编程大多不成立。逐条适配判断把这些挡掉了，避免返工。
